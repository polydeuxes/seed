# Constitutional Capability / Use Boundary Investigation

Repository authority wins.

## Bounded question

Across recurring repository evidence, when Seed receives or produces a claim that a system, actor, source, product, institution, or deployment has the capability to do something, what prevents Seed from treating that capability as proof that the capability was actually used without separately preserved event evidence, deployment evidence, actor evidence, temporal provenance, source authority, claim form, support path, confidence limits, Unknowns, and lawful stop?

This is exactly one bounded constitutional investigation of:

```text
Capability != Use
```

It is not world research, not legal research, not implementation, not a technology-evaluation engine, not a vendor-claim engine, not a surveillance-evaluation engine, not a harm engine, not a causation engine, and not a capability or use-event schema.

## Capability analysis

Repository evidence does not support a general constitutional ontology of `capability` for public-world technologies, vendors, governments, products, surveillance tools, security tools, or automation. App-visible knowledge reachability did not preserve the exact breadcrumb `Capability != Use` as repository knowledge, and source navigation did not expose projected source-fact support for the combined capability/use query.

The smaller supported meaning is conditional and source-bounded:

```text
capability evidence = preserved support that some source, artifact, implementation surface, setting, documentation, observation, or diagnostic says a subject can, supports, may, is allowed to, or is configured to do X, within a bounded source role, deployment scope, time, confidence, and claim form.
```

That meaning is weaker than actual use. It can support only a capability-shaped claim unless separate evidence supports use, event occurrence, actor conduct, temporal occurrence, lawful authority, causation, harm, proof, or implementation pressure.

Capability-adjacent repository implementation vocabulary also exists in local operational contexts, such as current capabilities, capability candidates, verification evidence, promotion readiness, capability needs, and capability pressure. Those surfaces are not a public-world capability/use model. They are Seed-local operational and diagnostic views whose authority remains bounded by their implementation declarations, event/recording behavior, support paths, and non-mutation boundaries.

## Use analysis

Repository evidence likewise does not support a general public-world use-event ontology. The recurring smaller meaning is:

```text
use = an event- or conduct-shaped claim that a capability, function, artifact, actor, deployment, or system actually performed or invoked X in a bounded deployment, time, actor/source context, and support path.
```

Use requires more than capability evidence. It requires separately preserved event evidence or conduct evidence sufficient for the exact claim form. Depending on the claim, that may include logs, traces, operator action records, device/system events, deployment-specific configuration history, temporal provenance, actor attribution, source authority, support path, confidence, Unknown preservation, and lawful stop.

`Use` is not automatically lawful use, causal effect, harmful use, proof of conduct, or implementation pressure. Those are stronger claim forms requiring stronger and different support.

## Can-do / did-do distinction

`can do X` is modal, dispositional, potential, supported-feature, available-function, documented, authorized, permitted, or vendor-asserted language. It may be true even when X never happened.

`did X` is event, occurrence, conduct, or history language. It claims that X happened in a particular setting, often at or within a particular time, through an actor or system path.

Repository evidence blocks the promotion from `can` to `did` through recurring non-collapse boundaries:

- observation is not truth or event occurrence;
- evidence is not claim-strength escalation;
- model output is not evidence;
- source role is not general authority;
- repetition is not proof;
- narrative is not history;
- legal authority is not warrant;
- absence of contrary evidence is not proof;
- translation/rendering is not source equivalence;
- presentation vocabulary is not repository knowledge;
- diagnostic visibility is not cluster mutation or fact promotion;
- Unknown preservation is a lawful result.

Therefore `can do X` can support at most `capability X is claimed/supported/available/authorized/documented within boundary B` unless use evidence separately supports `X was used`.

## Vendor-claim / deployment-use distinction

A vendor-claimed capability is a source-attributed claim from a vendor or vendor-controlled artifact. It may support claims such as:

```text
Vendor V claimed product P can do X in source S at time/version T.
```

It does not, without more, support:

```text
Deployment D used X.
```

Deployment use requires deployment-specific evidence: installed version, configuration, enabled state where relevant, logs or telemetry showing invocation, deployment boundary, actor/system path, temporal provenance, source authority for that deployment, and confidence limits. A vendor has source-role authority for its own statement, documentation, marketing, or contract language only within preserved boundaries. The vendor claim is not automatically an observation from the deployment, not a deployment event, not independent source support, and not proof of conduct.

## Documented-feature / enabled-feature distinction

A documented feature means a document describes, advertises, specifies, explains, or permits the feature. Its source role is documentation evidence. It can support a source-attributed documentation claim when source, version, date, product, and claim form are preserved.

An enabled feature means a deployment, system, account, policy, or configuration state had the feature turned on or otherwise active under a preserved state boundary. It requires settings, configuration, administrative records, runtime state, deployment inventory, or equivalent state evidence tied to the specific deployment and time.

Documentation alone does not prove enablement because documentation may cover optional, unavailable, disabled, future, deprecated, tier-limited, jurisdiction-limited, version-limited, or non-deployed behavior.

## Enabled-feature / actual-use distinction

An enabled function may be ready for invocation, scheduled, available to an actor, or active in configuration. It is still not proof that it ran, was invoked, affected a subject, produced output, or caused a result.

Actual use requires event-shaped support: invocation log, transaction, query, alert, access record, runtime trace, operator action, generated output tied to invocation, observed side effect with support path, or other competent event evidence. Enablement may support a bounded precondition claim:

```text
X was enabled in deployment D during time window T.
```

It does not support the stronger event claim:

```text
X was used during T.
```

unless event evidence separately bridges that gap.

## Authorized-use / actual-use distinction

Authorized capability or authorized use is permission, authority, entitlement, policy allowance, legal permission, contract scope, role assignment, warrant/approval, or administrative ability. It is authority/permission evidence, not conduct evidence.

Actual use is occurrence. A person or system may be authorized and not use; may use outside authorization; may be authorized only for a subset of actions; or authorization may be expired, jurisdiction-limited, procedurally defective, or Unknown.

Repository legal-authority/warrant evidence reinforces this boundary: legal authority or policy permission is not constitutional warrant and not proof of actual conduct. Authorization can support a bounded claim that use would have been permitted under preserved conditions only if the authority, jurisdiction, time, scope, actor, and claim form are preserved. It cannot prove use happened.

## Actual-use / lawful-use distinction

Actual use says X happened. Lawful use says X happened and was lawful under a competent authority boundary. Those are different claim forms.

Lawful use requires actual-use evidence plus legal or policy authority evidence, source-of-law or source-of-authority lineage where relevant, jurisdiction, temporal authority, scope, actor authorization, procedural posture, negative authority, limitations, confidence, Unknowns, and lawful stop. Actual use by itself may be lawful, unlawful, authorized, unauthorized, accidental, operationally permitted, contractually permitted, constitutionally warranted, or Unknown.

Seed may not promote `used` into `used lawfully` merely because a policy allowed use, a statute existed, a vendor contract permitted use, an operator believed use was allowed, or a court/source said something related. The lawful-use claim needs its own preserved warrant path.

## Use / causation distinction

Use is occurrence or invocation. Causation is a relationship between the use and an effect. Repository evidence on claim strength, support paths, narrative/history, repetition/proof, and lawful reliance blocks converting one event into causal explanation without separately preserved causal support.

A use-related record may be temporally adjacent to a result, but temporal adjacency is not causation. Causation requires support appropriate to the claim: mechanism, sequence, counterfactual or dependency evidence where relevant, exclusion of alternate causes where the claim requires it, source authority, confidence, and preserved Unknowns.

Seed may say `X was used` only when use evidence supports it. Seed may say `X caused Y` only when the support path also establishes the causal relation at the stated strength.

## Use / harm distinction

Use is not harm. Harm is an injury, adverse effect, rights impact, operational damage, loss, deprivation, privacy invasion, security compromise, or other consequence claim. Harm may require affected-subject evidence, impact evidence, legal or policy definitions, temporal linkage, causation, scope, severity, confidence, and Unknown preservation.

A capability may be harmful in design, risky, sensitive, or capable of harmful uses without having harmed anyone. A use may occur without proven harm. A harm may occur without proof that the capability caused it. Seed must preserve these boundaries separately.

## Feature-availability / event-occurrence distinction

Feature availability is state or possibility: the feature was included, accessible, licensed, installed, configured, or present. Event occurrence is history: something happened.

Availability can be evidence for bounded readiness or possibility, and sometimes for opportunity. It is not event evidence. Repository event/history/provenance boundaries require occurrence claims to preserve event source, time, actor/system path where relevant, and support. A feature existing in a deployment is not an event of that feature being used.

## Source-role analysis for capability/use claims

Different materials have different source roles. Their role determines the lawful claim form they can support.

### Logs

Logs may support event occurrence, system state, invocation, access, error, or audit trail claims if source identity, log generation path, time, scope, retention, integrity, parsing, and deployment boundary are preserved. Logs can still be incomplete, ambiguous, transformed, sampled, delayed, absent, spoofed, misconfigured, or outside the relevant scope. A log showing a feature was available is not automatically a log showing use; a log showing an event is not automatically lawful use, harm, or causation.

### Settings and configuration

Settings may support enabled/disabled, configured, permitted, installed, or deployment-state claims. They do not by themselves prove invocation or use. Settings need deployment identity, effective time, version, inheritance/override path, actor authority for changes where relevant, and confidence.

### Documentation

Documentation may support documented-feature, described behavior, product claim, configuration instruction, or intended-use claims. It does not prove a deployment had that product/version, that a feature was enabled, that an actor used it, that use was lawful, or that a result occurred. Documentation may be versioned, aspirational, incomplete, optional, marketing-oriented, or inapplicable to the deployment.

### Policies

Policies may support permitted, prohibited, authorized, required, or intended governance claims. They are not event evidence. They do not prove compliance, noncompliance, actual use, lawful use under external law, causation, harm, or technical capability unless separately supported.

### Contracts

Contracts may support party obligations, permissions, service descriptions, warranties, disclaimers, scope, procurement terms, or source-attributed representations. They do not prove actual deployment configuration, actual use, lawful use, technical truth, performance, harm, or causation without separate evidence. Contractual permission is not conduct.

### Vendor claims

Vendor claims may support source-attributed vendor assertion, marketed capability, product description, roadmap, support statement, or feature representation. They are not independent deployment evidence, not proof of actual use, not proof that the feature worked as claimed, and not legal warrant.

### Operator notes

Operator notes may support source-attributed human report, intent, recollection, procedure, explanation, or local observation. They require source identity, role, time, context, basis of knowledge, and confidence. Notes can be orientation evidence but are not automatically logs, settings, independent observations, event records, lawful authority, causal proof, or harm evidence.

## Event-evidence / deployment-evidence / actor-evidence analysis

Capability/use separation depends on preserving which boundary a piece of evidence addresses:

- **Event evidence** supports that something happened. It needs event source, event type, time or time window, subject/object, system path, record origin, integrity/translation boundary, and confidence.
- **Deployment evidence** supports where a product, system, feature, version, setting, account, or function existed. It needs deployment identity, environment, version, configuration, scope, effective time, and source authority for that deployment.
- **Actor evidence** supports who or what acted, had access, invoked, authorized, configured, or controlled. It needs actor identity, role, authority, authentication/attribution path, system account mapping, delegation, time, and confidence.

Use-related claims often require all three. A deployment may have capability without actor use. An actor may be authorized without event occurrence. An event may occur without proving who caused it. Missing any required boundary leaves Unknowns.

## Temporal-provenance analysis

Time is not decorative. Capability, availability, enablement, authorization, use, lawful use, causation, and harm can each have different effective times.

Seed must preserve, as applicable:

- document publication/version time;
- vendor claim time;
- product/version availability time;
- deployment installation or configuration time;
- setting effective time;
- authorization grant/expiration time;
- actor access time;
- event occurrence time;
- observation/collection time;
- record creation/retention time;
- legal/policy authority time;
- harm/effect time.

A capability existing at one time does not prove it existed at another. Authorization at one time does not prove use at that time. Use at one time does not prove lawful authority at that time. Temporal gaps require Unknown preservation or narrower claims.

## Claim-form analysis

Claim form controls support requirements. Repository evidence repeatedly blocks claim-strength escalation. Distinct claim forms include:

- `source S says product P can do X`;
- `documentation D describes feature X`;
- `deployment D included capability X`;
- `feature X was available in deployment D`;
- `feature X was enabled in deployment D during T`;
- `actor A was authorized to use X during T`;
- `actor/system A used X in deployment D at T`;
- `use of X was lawful under authority Y`;
- `use of X caused result R`;
- `use of X harmed subject H`;
- `evidence proves conduct C`;
- `repository implementation should change because of X`.

Evidence for one claim form does not automatically satisfy another. The lawful result is the strongest claim form actually supported by preserved evidence.

## Confidence / evidence-strength analysis

Confidence must track the preserved support path and claim strength. High confidence may be lawful for narrow claims such as `the app-visible source-navigation query found no projected support facts` or `a specific document says vendor V claimed X`. Lower confidence or Unknown is required when evidence is indirect, transformed, source-attributed only, incomplete, temporally mismatched, deployment-mismatched, actor-mismatched, dependent, repeated, or outside source authority.

Capability evidence can raise confidence that a capability was possible, documented, supported, available, or authorized. It does not raise confidence in actual use beyond what the event/use evidence supports. Confidence cannot bridge a missing support path.

## Unknown preservation

When evidence does not separately preserve the required boundary, Seed must preserve Unknowns rather than promote capability into use. Relevant Unknowns include:

- whether capability X existed at all;
- whether X was supported by product/version P;
- whether X was available in deployment D;
- whether X was enabled during time window T;
- whether actor A had access or authorization;
- whether X was actually invoked;
- whether a recorded event corresponds to X;
- whether X was used against subject S;
- whether use was lawful;
- whether use caused result R;
- whether harm H occurred;
- whether X caused harm H;
- whether source S had authority for the claim form;
- whether sources are independent;
- whether temporal provenance matches;
- whether transformed/rendered evidence preserves source meaning;
- whether confidence is sufficient for the stronger claim;
- whether implementation pressure exists.

Unknown preservation is a lawful stop, not a failed investigation.

## Required analysis answers

### 1. What does repository evidence already mean by capability, if anything?

Only a smaller, bounded meaning is supported: a source-, artifact-, implementation-, setting-, or documentation-backed claim that something can, supports, permits, or is configured to do X within preserved scope. Repository evidence does not establish a general public-world capability ontology.

### 2. What does repository evidence already mean by use, if anything?

Use means an event- or conduct-shaped occurrence claim only when separately supported. Repository evidence does not establish a general use-event schema, but it does distinguish observation, evidence, fact support, event/history/provenance, claim form, and support path.

### 3. What distinguishes `can do X` from `did X`?

`Can do X` is capability/possibility/support/permission language. `Did X` is event/history/conduct language. The distinction is preserved by event evidence, temporal provenance, actor/deployment evidence, claim form, support path, confidence limits, and Unknowns.

### 4. What distinguishes vendor-claimed capability from deployment use?

Source role and boundary. Vendor claim is vendor-source evidence about a representation. Deployment use requires deployment-specific and event-specific evidence.

### 5. What distinguishes documented feature from enabled feature?

Documentation describes or claims a feature. Enablement requires deployment-specific settings, configuration, runtime state, or administrative evidence for a time and scope.

### 6. What distinguishes enabled feature from actual use?

Enablement is state/readiness. Actual use is occurrence/invocation. Logs, traces, actions, events, or equivalent event evidence are required for the latter.

### 7. What distinguishes authorized use from actual use?

Authorization is permission or authority. Actual use is conduct. Permission does not prove conduct.

### 8. What distinguishes actual use from lawful use?

Lawful use requires actual use plus a competent legal/policy/warrant path within jurisdiction, time, scope, actor authority, negative authority, and lawful stop.

### 9. What distinguishes use from causation?

Use is occurrence. Causation is a supported relation between occurrence and effect. Causal claims require mechanism/support beyond occurrence.

### 10. What distinguishes use from harm?

Use is invocation/conduct. Harm is an adverse effect or injury claim requiring affected-subject, impact, legal/policy, temporal, causal, and confidence support as applicable.

### 11. What distinguishes feature availability from event occurrence?

Availability is state or possibility. Event occurrence is history. Availability may be a precondition but is not an occurrence record.

### 12. What distinguishes logs, settings, documentation, policies, contracts, vendor claims, and operator notes as source roles for capability/use claims?

Logs can be event/state records; settings can be configuration state; documentation can be described-feature evidence; policies can be permission/governance evidence; contracts can be obligation/representation evidence; vendor claims can be source-attributed product assertions; operator notes can be human reports or orientation. None is universally authoritative for every claim form.

### 13. What role do event evidence, deployment evidence, actor evidence, temporal provenance, source authority, claim form, support path, confidence, and Unknown preservation play?

They prevent claim-strength escalation. Each tells Seed which boundary is supported, which boundary is missing, how strong the claim may be, and where lawful stop is required.

### 14. Under what conditions, if any, can capability evidence lawfully support a bounded use-related claim?

Capability evidence can lawfully support weaker use-related claims such as possibility, availability, opportunity, prerequisite, documentation, permission, or source-attributed representation when the claim form says so. It can also be one component of a use claim when combined with independent event/deployment/actor/temporal evidence. Capability evidence alone cannot prove actual use.

### 15. What must be preserved before Seed may lawfully say `capability X existed in this system/deployment/time window`?

Seed must preserve system/deployment identity, capability identity, product/version or implementation path, source material, source role/authority, configuration or availability evidence, effective time window, support path, confidence, transformations, negative/limiting evidence where known, and Unknowns.

### 16. What must be preserved before Seed may lawfully say `capability X was used`?

Seed must preserve capability identity, deployment boundary, event evidence of invocation/operation, actor or system attribution where the claim requires it, temporal provenance, source authority, claim form, support path, confidence, transformation boundaries, contradictory/negative evidence where known, and Unknowns.

### 17. What must be preserved before Seed may lawfully say `capability X was used lawfully`, `caused harm`, or `produced result Y`?

For lawful use: actual-use evidence plus competent authority/warrant/policy/legal-source lineage, jurisdiction, time, scope, actor authorization, procedural conditions, negative authority, confidence, and Unknowns. For causation/result: actual-use evidence plus mechanism, sequence, dependency, result evidence, temporal relation, alternative-cause limits, confidence, and Unknowns. For harm: actual-use and causal evidence plus affected-subject/impact evidence, harm definition/scope, severity where claimed, confidence, and Unknowns.

### 18. What lawful stop condition prevents capability from becoming use, lawful use, harm, causation, proof of conduct, or implementation pressure?

If separate preserved evidence does not support the stronger claim form, Seed must stop at the bounded capability/source/availability/authorization claim and preserve the missing use, lawful-use, harm, causation, conduct-proof, or implementation-pressure boundary as Unknown.

### 19. Does this investigation create implementation pressure?

No. This investigation creates investigation pressure only. It identifies a boundary discipline and no concrete repository failure, failing diagnostic, failing test, missing implemented surface, or required schema.

### 20. Does repository evidence support implementation work yet?

No. Repository evidence supports no implementation work from this investigation. Implementing a capability/use model, vendor-claim engine, use-event schema, legal engine, harm engine, causation engine, or source-scoring system would exceed the evidence reviewed.

## Lawful bounded uses of capability evidence

Capability evidence may lawfully support:

- source-attributed claims: `source S claims X`;
- documentation claims: `document D describes X`;
- product/version claims where source support is competent;
- deployment availability claims where deployment evidence is preserved;
- enabled-state claims where configuration/effective-time evidence is preserved;
- authorization or permission claims where authority/scope/time evidence is preserved;
- possibility or opportunity claims stated as possibility/opportunity, not occurrence;
- prerequisite analysis for a separately supported use claim;
- Unknown preservation when use evidence is missing.

Capability evidence may not, by itself, support actual use, deployment use, actor use, event occurrence, lawful use, harm, causation, proof of conduct, or implementation pressure.

## Lawful stop

The lawful stop is:

```text
Capability evidence stops at capability-shaped claims unless separately preserved evidence supports the stronger use-shaped, lawful-use-shaped, causal, harm, conduct-proof, or implementation-pressure claim.
```

If the support path preserves only `can`, `supports`, `available`, `enabled`, `documented`, `vendor-claimed`, `authorized`, or `possible`, Seed must not output `did`, `used`, `used lawfully`, `caused`, `harmed`, `proved conduct`, or `requires implementation`. The missing stronger claim remains Unknown.
