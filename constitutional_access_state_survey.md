Repository authority wins.

## Boundary

This is exactly one bounded Constitutional Access State Survey. It determines only the recurring constitutional access surfaces that presently exist in repository evidence and the constitutional access state each begins in.

It does not recover implementation, security architecture, permission systems, runtime policy, provider execution, host management, or any new authorization architecture. It does not infer capability from access. It does not infer external grammar from access.

## Survey method

Each recurring access surface is surveyed independently before comparison. The survey asks only:


## Independently surveyed access surfaces

### 1. Operator-prose / inquiry-note access surface

**What external surface exists?**

Operator prose may appear as user-supplied text, pressure, prompt content, runtime user input, or a preserved inquiry note depending on the implemented local path. The smallest recurring example is the `.` path: a non-empty raw inquiry note may be preserved outside the event ledger.

**What constitutional responsibility does this surface enable?**

It enables Seed to truthfully acknowledge that operator-supplied material exists for a bounded local responsibility and, where supported, to orient attention over preserved note text and already projected repository material.

**What constitutional state does it begin in?**

It begins **uncommitted / preserved-external-representation only**. It is not an Observation, Evidence, Fact, repository attachment, command, plan, goal, semantic interpretation, or ownership claim merely because it was supplied.

**What lawful transition changes that state?**

A lawful transition occurs only when an implemented local responsibility preserves, inspects, or records the material with bounded provenance and then a further supported responsibility creates an observation, orientation view, or evidence-bearing result without overclaiming its truth.

**What neighboring access surfaces recur?**

- Runtime user-message intake.
- Inquiry-note preservation.
- Inquiry Orientation.
- Repository orientation.
- Event-ledger response events.
- Decision-facing context composition.

**Access != ?**

Access to operator prose is not capability, execution permission, source authentication, external grammar, semantic authority, or truth.

**What explicitly must this surface never imply?**

It must never imply that the operator's words are verified, that a command should execute, that Seed owns a goal or plan, that cluster truth changed, that repository knowledge was promoted, or that an authorization context exists.

### 2. File / repository artifact access surface

**What external surface exists?**

File-backed material may appear as raw file bytes, bounded file samples, paths, sizes, hashes, extensions, declared purpose, repository source files, Markdown records, or JSON/YAML/INI candidate content.

**What constitutional responsibility does this surface enable?**

It enables bounded inspection, candidate format detection, provenance preservation, repository orientation, and, when a provider/source contract supports it, transformation into observations or relationship evidence.

**What constitutional state does it begin in?**

It begins **candidate / uncommitted external representation**. Raw file access is not repository attachment and not repository truth.

**What lawful transition changes that state?**

A lawful transition occurs when bounded inspection or provider-specific validation accepts a supported shape and constructs a valid observation, source relationship, or preserved candidate record. Unsupported formats, malformed records, no-match states, and unknowns stop locally.

**What neighboring access surfaces recur?**

- Input inspection.
- JSON observation import.
- Ansible inventory parsing.
- Repository-source observation.
- Inquiry/document orientation.
- Observation Agreement and grammar observation.

**Access != ?**

File access is not provider capability, repository attachment, source truth, grammar ownership, execution authority, or semantic understanding.

**What explicitly must this surface never imply?**

It must never imply that arbitrary bytes are trusted, that embedded references are resolved, that content is executed, that JSON syntax is enough for observation, that repository vocabulary is promoted, or that repository files automatically become constitutional law.

### 3. Provider payload / observation-source access surface

**What external surface exists?**

Provider material recurs as Ansible inventory text, Prometheus HTTP metric responses, systemd JSON rows, dpkg status text, JSON observation payloads, local platform readings, and repository source observations.

**What constitutional responsibility does this surface enable?**

It enables bounded provider/source translation into observations with provenance, source metadata, confidence limits, and local shape validation.

**What constitutional state does it begin in?**

It begins **provider-representation unaccepted / shape-uncommitted** until a specific source contract accepts the representation. After acceptance, it may become **observation-bearing**, not automatically fact-bearing or verified.

**What lawful transition changes that state?**

A lawful transition occurs through provider-specific acceptance: supported parser selection, response-shape validation, required-field validation, allowlisted query families, valid metadata, and successful Observation construction.

**What neighboring access surfaces recur?**

- File/input inspection.
- Observation ingestion.
- Local-host observation.
- Prometheus/source adapters.
- Systemd and package observation.
- Event-ledger ingestion.
- Projection and state views.

**Access != ?**

Provider access is not arbitrary query freedom, provider management, network reachability proof, execution capability, capability promotion, or verification.

**What explicitly must this surface never imply?**

It must never imply that provider data is true merely because it arrived, that malformed payloads should partially promote, that observation equals fact, that provider labels identify canonical entities without supported evidence, or that Seed may mutate the provider.

### 4. Local-host bounded observation access surface

**What external surface exists?**

The local-host surface recurs as read-only/local-only access to identity, hosts-file and resolver material, host description, network metadata, mounts/filesystems, storage topology, listeners, users/groups, packages, and systemd observations.

**What constitutional responsibility does this surface enable?**

It enables bounded local observation of host-scoped domains while preserving reader, vantage point, source, and provenance.

**What constitutional state does it begin in?**

It begins **locally observable / read-only / non-mutating**. It is observation access, not host-control access.

**What lawful transition changes that state?**

A lawful transition occurs when local observer methods or delegated source classes emit bounded observations that later participate in event history, projection, state views, or context views. Further promotion requires evidence-backed fact/support processing.

**What neighboring access surfaces recur?**

- Filesystem/mount observation.
- Storage topology observation.
- Listener observation.
- User/group observation.
- Package observation.
- Systemd observation.
- Host observation domains and reader/transport boundaries.

**Access != ?**

Local-host access is not SSH access, remote host access, privilege proof, login activity, sudo authority, service ownership, availability, reachability, health, repairability, or management capability.

**What explicitly must this surface never imply?**

It must never imply that Seed may execute shell commands, mutate the host, validate connectivity, infer account safety, infer external accessibility, assert filesystem health, assert service/application ownership, or manage local resources.

### 5. Runtime interaction / response access surface

**What external surface exists?**

Runtime interaction recurs as local CLI input, user messages, model decisions, answer/question/refusal responses, tool-need requests, registered tool-call routing, and state-patch proposal handling.

**What constitutional responsibility does this surface enable?**

It enables bounded interaction routing to owner services and visible response behavior while preserving distinctions among answer composition, request recording, capability gap resolution, registered operation execution, and state-patch mutation.

**What constitutional state does it begin in?**

It begins **interaction-visible / authority-uncommitted**. A message or decision may be routed only according to existing implementation boundaries; it does not grant execution authority or source authority by appearing.

**What lawful transition changes that state?**

A lawful transition occurs when validated runtime decisions are routed to their existing owners, such as response recording, tool-need recording, provider recommendation, registered operation execution after policy evaluation, or state-patch proposal handling.

**What neighboring access surfaces recur?**

- Operator-prose intake.
- Source authority metadata.
- Tool need and provider recommendation.
- Registered operation execution.
- Tool execution policy.
- Pending action lifecycle.
- Event-ledger response and tool-call events.

**Access != ?**

Runtime interaction access is not capability ownership, behavior selection, planning, provider execution, source authentication, or authorization architecture.

**What explicitly must this surface never imply?**

It must never imply that user input is authenticated, that request_tool executes, that observation implies management, that answer composition mutates cluster state, that policy vocabulary creates runtime authority, or that legacy execution-authorization metadata is core behavior.

### 6. Diagnostic / audit / operational visibility access surface

**What external surface exists?**

Diagnostic and audit surfaces recur as inventory reports, shape audits, reasoning-path audits, observation-domain reports, integrity summaries, selection-path audits, state summaries, and other read-only visibility outputs.

**What constitutional responsibility does this surface enable?**

It enables Seed to make existing visibility, gaps, unknowns, unsupported surfaces, evidence paths, inventory rows, and read-only diagnostic findings inspectable.

**What constitutional state does it begin in?**

It begins **read-only diagnostic visibility / non-mutating**. It may expose evidence or readiness status but does not itself become cluster truth, fact truth, runtime authority, or implementation recovery.

**What lawful transition changes that state?**

A lawful transition occurs only if a separate implementation-backed owner records, validates, or promotes a finding under its own evidence rules. Recording diagnostic output must preserve diagnostic-run scope and read-only boundaries unless an intentionally different boundary is explicitly supported.

**What neighboring access surfaces recur?**

- Diagnostic inventory.
- Diagnostic shape audit.
- Observation permission/domain reports.
- Reasoning-path audit.
- Projection integrity summary.
- State summary and read-only views.
- Event-ledger recording when explicitly supported.

**Access != ?**

Diagnostic access is not cluster mutation, proof of correctness, verification, policy enforcement, runtime autonomy, or fact promotion.

**What explicitly must this surface never imply?**

It must never imply that diagnostic findings silently become cluster truth, that recordable output mutates runtime entities, that visibility is authority, that unknowns are failures, or that readiness equals permission to proceed.

### 7. Event-history / ledger access surface

**What external surface exists?**

Event history recurs as append-only event records, observation-ingestion events, response events, tool-call lifecycle events, pending-action events, and historical inputs used for projection and audit.

**What constitutional responsibility does this surface enable?**

It enables durable, replayable preservation of what Seed recorded as having happened within its authority boundary so projections can be rebuilt, inspected, challenged, and explained.

**What constitutional state does it begin in?**

It begins **append-only record / history-bearing**, not objective reality and not current truth by itself.

**What lawful transition changes that state?**

A lawful transition occurs when recorded events are projected into current state or interpreted by supported evidence/fact/projection responsibilities. Current lawful condition remains distinct from event history.

**What neighboring access surfaces recur?**

- Observation ingestion.
- Projection store.
- State and context views.
- History brief.
- Runtime response/tool-call recording.
- Diagnostic recording boundaries.

**Access != ?**

Event-history access is not complete memory, objective reality, verification, current state, cached projection, mutation authority, or external testimony truth.

**What explicitly must this surface never imply?**

It must never imply that recording testimony makes it true, that projections are source-of-truth, that history authorizes execution, that conflicts are erased, or that all external reality is captured.

### 8. Projected State / State View / Context View access surface

**What external surface exists?**

Projected access recurs as State, State Views, Context Views, State Summary, storage/filesystem projections, availability summaries, graph issue visibility, contradiction/confidence/evidence layers, and decision-facing context.

**What constitutional responsibility does this surface enable?**

It enables deterministic read-only access to already projected state and evidence layers for inspection, orientation, context packaging, and answer support.

**What constitutional state does it begin in?**

It begins **projected / read-only / derived-current-view**. It is not source-of-truth, not observation acquisition, and not verification.

**What lawful transition changes that state?**

A lawful transition occurs upstream through event/observation/fact support changes and projection rebuild. Views themselves do not mutate state, append events, invoke providers, or execute behavior.

**What neighboring access surfaces recur?**

- Event history.
- Observation and fact support.
- State Summary.
- Context Views.
- Inquiry Orientation.
- Storage/filesystem lens-like projections.
- Decision-facing answer composition.

**Access != ?**

Projected access is not provider access, acquisition access, current external reality, verification, mutation, execution, or universal orientation.

**What explicitly must this surface never imply?**

It must never imply that cache equals truth, that presentation labels become knowledge, that a lens is an observation source, that decision context may invent missing facts, or that projection-only interpretations are topology truth.

### 9. Service / listener visibility access surface

**What external surface exists?**

Service and listener visibility recurs through local socket tables, systemd unit observations, service availability predicates, endpoint scrape availability, and projected host/service/entity summaries.

**What constitutional responsibility does this surface enable?**

It enables bounded visibility into locally reported listener/unit/service states and projected service availability claims when supported by observations.

**What constitutional state does it begin in?**

It begins **visible-as-observed / operationally uncommitted**. A listener, unit, or service datum may be observed, but management, reachability, ownership, and health remain uncommitted unless separately supported.

**What lawful transition changes that state?**

A lawful transition occurs when a bounded observation source emits valid observations and later evidence/fact/projection layers support specific availability, identity, or relationship claims.

**What neighboring access surfaces recur?**

- Local-host listener observation.
- Systemd observation.
- Prometheus scrape/metric observations.
- Host/service projection.
- Availability summaries.
- Context and state views.

**Access != ?**

Service/listener access is not external accessibility, responsiveness, process ownership, application ownership, management authority, repair authority, or runtime execution.

**What explicitly must this surface never imply?**

It must never imply that a local listener is reachable externally, that a systemd unit is safe or healthy, that Seed can start/stop/restart services, that scrape availability verifies service correctness, or that service visibility grants operational control.

## Comparison

### Recurring access surfaces

Recurring constitutional access surfaces recovered by repository evidence are:

1. Operator-prose / inquiry-note access.
2. File / repository artifact access.
3. Provider payload / observation-source access.
4. Local-host bounded observation access.
5. Runtime interaction / response access.
6. Diagnostic / audit / operational visibility access.
7. Event-history / ledger access.
8. Projected State / State View / Context View access.
9. Service / listener visibility access.

### Recurring initial access states

The recurring initial states are:

- **Uncommitted / Null-protected** for subjects not yet moved by evidence.
- **Preserved external representation only** for operator prose and other raw external material.
- **Candidate / shape-uncommitted** for files and provider payloads before accepted contracts.
- **Observation-bearing only** after valid observation construction, without automatic fact or verification promotion.
- **Read-only / non-mutating** for local-host observations, diagnostics, views, and context surfaces.
- **Append-only history-bearing** for ledger events, without objective-reality or current-truth authority by itself.
- **Projected / derived-current-view** for State, State Views, Context Views, and summaries.
- **Operationally uncommitted visibility** for service/listener surfaces.

### Recurring lawful access transitions

Recurring lawful transitions are evidence-bound and local:

- Null or uncommitted subject to bounded presence when an existing local responsibility preserves or inspects an external representation.
- Candidate external representation to accepted observation only through provider/source-specific validation.
- Observation to fact/support/projection only through existing evidence and fact-support responsibilities.
- Event history to current projection through projection rebuild.
- Projection to view/context/orientation through deterministic read-only rendering or packaging.
- Diagnostic finding to any stronger status only through a separate owner with explicit evidence rules.
- Runtime message/decision to behavior only through validated routing to existing owner services.

No recurring transition supports automatic movement from access to capability, authorization, external grammar ownership, runtime policy, or mutation.

### Recurring neighboring access surfaces

The surfaces recur in neighborhoods rather than one universal access system:

- External representation, inquiry note, input inspection, and repository orientation.
- Provider parsing, observation sources, observation ingestion, and event ledger.
- Local-host domain readers, systemd/package/listener/filesystem observations, and host observation domains.
- Event ledger, projection, State Views, Context Views, State Summary, and Inquiry Orientation.
- Runtime routing, tool need, provider recommendation, policy-adjacent vocabulary, registered operation execution, and pending action lifecycle.
- Diagnostic inventory, diagnostic shape audit, reasoning-path audit, observation-domain report, and integrity summary.

### Recurring access boundaries

Repository evidence repeatedly preserves these boundaries:

- Access is not capability.
- Access is not external grammar.
- Access is not authorization architecture.
- Visibility is not authority.
- Observation is not fact.
- Provider response is not verification.
- File syntax is not repository attachment.
- Event history is not objective reality.
- Projection is not source-of-truth.
- Diagnostic output is not cluster truth.
- Runtime interaction is not execution permission.

### Preserved unknowns

- The repository does not recover one universal constitutional access object.
- The repository does not recover one universal first access transition.
- The repository does not prove that all possible external participants have the same source authority.
- The repository does not establish browser, API, voice, webcam, automation, SSH-origin, shell-user, or remote-origin identity as a general current access layer.
- The repository does not establish arbitrary provider query access.
- The repository does not establish remote host control or management access.
- The repository does not establish that presentation vocabulary is repository knowledge.
- The repository does not establish that readiness is authority to proceed.
- The repository does not establish a general runtime access architecture.
