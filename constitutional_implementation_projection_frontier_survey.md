### 2. Pressure is not recovery

Recent pressure recovery distinguishes pressure from goal authority. The recovered transition `goal -> pressure` supports treating implementation comparison pressure as attention or inspection pressure rather than as an implementation mandate, priority engine, planning surface, or constitutional destination.

For this survey, implementation artifacts may create pressure because they are observable and can expose mismatch, compression, compatibility boundaries, or visibility gaps. That pressure does not by itself recover implementation projection or require code change.

The recurring transition `acceptance -> family-local admission` matters because implementation evidence may be accepted locally by a bounded family without being accepted as global constitutional authority.

An implementation slice can admit a local owner, helper, artifact, consumer, or compatibility boundary. That admission remains family-local and evidence-bound. It does not promote implementation vocabulary into repository-wide constitutional knowledge.

### 4. Work over time is durable boundary preservation

The recurring transition `work over time -> durable boundary preservation` is reinforced by implementation recovery slices: the repository repeatedly preserves behavior and compatibility while making one compressed boundary observable.

When constitutional grammar is projected onto implementation artifacts, the recurring constitutional work is not future-directed architecture synthesis. ### 5. Artifact availability is bounded reachability without promotion

The transition `artifact availability -> bounded reachability-without-promotion` directly constrains implementation evidence. An implementation artifact can be reachable, inspectable, consumed, emitted, or compared without becoming preserved constitutional knowledge or cluster truth.

This is especially important for rendered strings, CLI flags, diagnostic rows, source code symbols, projection stages, and event names. Their availability can support investigation, but does not automatically promote them.

### 6. Constitutional proprioception is grammar-bounded orientation

The transition `constitutional proprioception -> grammar-bounded orientation` supports using recovered grammar to orient inquiry about implementation without turning that orientation into a planning system, ontology, or autonomous self-modification loop.

## Supported distinctions

### Constitutional grammar

Constitutional grammar is the recovered interpretive constraint: evidence, provenance, authority boundary, negative authority, locality, distinction, attachment, lawful reliance, lawful acceptance, preserved Unknowns, confidence limits, and lawful stop.

It constrains interpretation of implementation evidence. It is not an implementation engine.

### Implementation artifact

An implementation artifact is repository-visible code, test, CLI surface, diagnostic registry row, projection stage, event name, helper, value object, file, or rendered output.

It can be observed and compared. It does not own constitutional authority merely by existing.

### Implementation evidence

Implementation evidence is what a repository artifact proves through code, tests, app output, documented slice behavior, or observable runtime/diagnostic boundary.

It can support a local recovery. It cannot silently override repository authority.

### Implementation comparison

Implementation comparison is the grammar-bounded act of asking whether an artifact's observed role matches or violates a recovered boundary.

It is investigative pressure. It is not a feature named implementation projection.

### Implementation divergence

Implementation divergence is a supported mismatch, compression, missing visibility, shape mismatch, unsupported consumer relation, or boundary gap shown by implementation evidence.

Divergence creates pressure for bounded investigation or a small implementation-backed change. It is not proof of constitutional failure unless authority and evidence support that conclusion.

### Implementation realization

Implementation realization is a code/test-backed local recovery that makes a supported boundary observable while preserving compatibility.

The reviewed evidence supports implementation realization only as local bounded recovery, not as automatic projection of constitutional grammar into code.

### Compatibility preservation

Compatibility preservation is the repeated implementation constraint that existing runtime behavior, public shape, event-ledger behavior, diagnostics, schema, CLI behavior, JSON output, cache behavior, and projection behavior remain unchanged unless the bounded task explicitly changes them.

### Lawful acceptance

Lawful acceptance is bounded and family-local. It admits only what evidence supports, preserves negative authority, and keeps unsupported candidate ideas Unknown.
