# Runtime vs RuntimeLoop reassessment

This reassessment updates the earlier parity inventories after the shared-service extractions and RuntimeLoop improvements. It focuses on behavior that remains relevant to whether `seed_runtime.runtime.Runtime` can be removed, not on infrastructure that is already shared.

## Shared capabilities

The following capabilities are now effectively shared through common services or shared domain boundaries and should not be treated as remaining Runtime-vs-RuntimeLoop gaps by themselves:

- **Tool recommendation lookup and ranking.** Both paths use `ToolRecommendationService` to turn a `ToolNeed` plus projected state into ranked provider recommendations. The service is read-only and owns the capability-catalog/ranker interaction, while each runtime still owns its own events and response shape. [`seed_runtime/tool_recommendations.py`](../seed_runtime/tool_recommendations.py) [`seed_runtime/runtime.py`](../seed_runtime/runtime.py) [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py)
- **Registered-tool existence, status, input-schema, and output-schema validation.** `ToolValidationService` centralizes lookup and schema validation. Runtime uses it through `DecisionValidator` and `ToolExecutor`; RuntimeLoop injects it into `ToolExecutionPolicyService` and also uses it for handler output validation. [`seed_runtime/tool_validation.py`](../seed_runtime/tool_validation.py) [`seed_runtime/decisions.py`](../seed_runtime/decisions.py) [`seed_runtime/execution.py`](../seed_runtime/execution.py) [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py)
- **Validation-before-policy sequencing.** `ToolExecutionPolicyService` now centralizes the sequence of resolving a registered tool, checking registration status, validating input, and then evaluating policy. Runtime and RuntimeLoop still interpret the returned validation/policy result differently, but the pre-execution ordering itself is shared. [`seed_runtime/tool_execution_policy.py`](../seed_runtime/tool_execution_policy.py)
- **Request-tool recommendation enrichment.** Both runtimes can create a tool need and include ranked recommendations for the requested capability. The remaining differences are event vocabulary and result contracts, not recommendation behavior. [`seed_runtime/runtime.py`](../seed_runtime/runtime.py) [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py)
- **Basic answer, request-tool, and call-tool decision coverage.** RuntimeLoop now covers the default API/CLI user-message path for answers, tool requests, policy-denied tool calls, handler failures, successful tool results, fact extraction from tool results, and decision journaling. [`seed_runtime/api.py`](../seed_runtime/api.py) [`scripts/seed_local.py`](../scripts/seed_local.py) [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py)
- **Provider failure handling in RuntimeLoop.** RuntimeLoop catches decision-provider exceptions, emits `runtime.decision.provider_failed`, records a decision-journal entry with `outcome="provider_failed"`, and returns a structured `RuntimeResult` error instead of letting provider exceptions escape. [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py)
- **Clarify/refuse semantic mapping documentation and behavior.** The shared intent builder intentionally maps clarify and refuse intents to RuntimeLoop `answer` decisions, while the legacy Runtime path still maps them to `ask_question` and `refuse`. This is now documented as an intentional semantic choice, not an accidental missing branch. [`seed_runtime/intent_classifier.py`](../seed_runtime/intent_classifier.py)
- **State projection and decision-ready context view for RuntimeLoop.** RuntimeLoop projects state through the projection-cache path and includes `DecisionContextView`, giving its providers canonical decision-oriented state access. The older `ContextPacket` remains a separate Runtime provider boundary. [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py) [`seed_runtime/context_views.py`](../seed_runtime/context_views.py)

## Runtime-only capabilities

These capabilities still exist only on the legacy `Runtime` user-message path:

- **Legacy `ContextPacket` provider boundary with retry prompts and context budgeting.** Runtime composes `ContextPacket` through `ContextComposer`, including section budgeting, open tool needs, evidence selected under facts, tool schemas, `decision_schema`, and retry prompts. RuntimeLoop uses a different `RuntimeContext` shape with `DecisionContextView`; it does not expose the legacy packet or retry-prompt contract. [`seed_runtime/context.py`](../seed_runtime/context.py) [`seed_runtime/runtime.py`](../seed_runtime/runtime.py)
- **Decision retry loop for model validation, tool-intent rejection, and parse failures.** Runtime retries up to `max_decision_retries`, records `model.decision.invalid`, `model.decision.intent_rejected`, and `model.decision.parse_failed`, and supplies corrected retry context. RuntimeLoop validates once and returns `malformed_decision` or `provider_failed` without asking the provider again. [`seed_runtime/runtime.py`](../seed_runtime/runtime.py) [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py)
- **`DecisionParseError`-specific handling.** Runtime distinguishes parse failures from ordinary invalid decisions, includes parse-failure classification metadata when present, and emits parse-failure events. RuntimeLoop's provider interface returns objects directly and only has generic provider-exception handling. [`seed_runtime/runtime.py`](../seed_runtime/runtime.py)
- **Tool-intent guard before routing tool calls.** Runtime applies `ToolIntentGuard` after decision validation and before routing, then can retry with an intent-specific correction prompt. RuntimeLoop has no equivalent behavior-level guard. [`seed_runtime/runtime.py`](../seed_runtime/runtime.py) [`seed_runtime/tool_intent.py`](../seed_runtime/tool_intent.py)
- **Distinct `ask_question` and `refuse` response contracts.** Runtime accepts `ask_question` and `refuse` decision kinds and emits `response.question` / `response.refusal` with `RuntimeResponse` kinds `question` / `refusal`. RuntimeLoop rejects those direct decision kinds and relies on answer-mapped clarify/refuse intent. [`seed_runtime/runtime.py`](../seed_runtime/runtime.py) [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py)
- **Model-proposed state patches.** Runtime accepts `propose_state_patch`, calls `StatePatchService`, emits state mutation events or `state.patch.rejected`, and returns `state_updated` or `invalid_state_patch`. RuntimeLoop's local `DecisionKind` excludes `propose_state_patch`, its validator rejects unsupported kinds, and it does not construct or call `StatePatchService`. [`seed_runtime/runtime.py`](../seed_runtime/runtime.py) [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py) [`seed_runtime/state_patches.py`](../seed_runtime/state_patches.py)
- **Pending-action creation for confirmation/approval and approved-action resume through `ToolExecutor`.** Runtime's tool execution path creates pending actions for `require_confirmation` / `require_approval` outcomes and can resume approved pending tool calls exactly once. RuntimeLoop collapses all non-allow policy outcomes into `runtime.policy.denied` / `policy_denied` and does not create or resume pending actions. [`seed_runtime/execution.py`](../seed_runtime/execution.py) [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py)
- **Registered operation import-path execution.** Runtime's `ToolExecutor` loads and executes the registered tool implementation from its toolkit operation import path. RuntimeLoop executes only explicitly supplied in-memory `tool_handlers` and records `runtime.tool.handler_missing` if one is not provided. [`seed_runtime/execution.py`](../seed_runtime/execution.py) [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py)
- **Legacy CLI `--plan` dependency.** The default CLI and HTTP paths use RuntimeLoop, but the CLI `--plan` path still calls `LocalSeedApp.run_legacy`, which calls `Runtime.handle_user_message`. [`scripts/seed_local.py`](../scripts/seed_local.py)

## RuntimeLoop-only capabilities

These capabilities exist only on RuntimeLoop or are materially richer there than in legacy Runtime:

- **Structured run result and decision journal for every RuntimeLoop branch.** RuntimeLoop returns `RuntimeResult` with run id, decision id, context hash, decision reason, outcome, policy status, tool result/error, and appended event ids; it appends decision-journal records for answer, request-tool, malformed decision, provider failure, policy denial, unknown/invalid tools, handler failure, and success. Runtime returns legacy `RuntimeResponse` without this journal contract. [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py) [`seed_runtime/decision_journal.py`](../seed_runtime/decision_journal.py)
- **RuntimeTrace / CLI trace integration.** RuntimeLoop's journal/event vocabulary is the basis for run trace reconstruction and CLI trace/why-run commands. The old Runtime path does not produce the same trace-oriented decision records. [`seed_runtime/runtime_trace.py`](../seed_runtime/runtime_trace.py) [`tests/test_runtime_trace.py`](../tests/test_runtime_trace.py) [`tests/test_cli_trace.py`](../tests/test_cli_trace.py)
- **Projection-store cache plumbing.** RuntimeLoop uses `project_state_with_cache` so a run can participate in projection-cache semantics. Runtime directly calls `StateProjector.project`. [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py) [`seed_runtime/projection_store.py`](../seed_runtime/projection_store.py) [`seed_runtime/runtime.py`](../seed_runtime/runtime.py)
- **`DecisionContextView` in provider context.** RuntimeLoop includes the canonical decision-oriented state view in `RuntimeContext`; Runtime's provider contract exposes the older packet sections instead. [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py) [`seed_runtime/context_views.py`](../seed_runtime/context_views.py)
- **Provider-failed outcome.** RuntimeLoop records provider exceptions as first-class run outcomes. Runtime only has special handling for `DecisionParseError`; other model exceptions are not converted into a structured runtime result by `handle_user_message`. [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py) [`seed_runtime/runtime.py`](../seed_runtime/runtime.py)
- **RuntimeLoop-specific tool execution event vocabulary.** RuntimeLoop uses `tool.result`, `tool.failure`, `runtime.tool.invalid`, `runtime.tool.unknown`, and `runtime.tool.handler_missing`; Runtime's `ToolExecutor` uses `tool.call.started`, `tool.call.completed`, `tool.call.failed`, `tool.policy.blocked`, and `tool.approval.required`. [`seed_runtime/runtime_loop.py`](../seed_runtime/runtime_loop.py) [`seed_runtime/execution.py`](../seed_runtime/execution.py)

## Remaining semantic gaps

Only behavior-level gaps are listed here. Shared lookup, validation, recommendation, and validation-before-policy infrastructure are intentionally excluded.

| Gap | Current difference | Migration effort | Why |
| --- | --- | --- | --- |
| Pending confirmation/approval and resume | Runtime creates pending actions for `require_confirmation` / `require_approval` and can resume approved tool calls; RuntimeLoop treats every non-allow policy outcome as denial and has no resume path. | **Significant** | This affects policy semantics, pending-action lifecycle events, result contracts, and idempotent resume behavior, not just routing. |
| Model-proposed state patches | Runtime routes `propose_state_patch` through `StatePatchService`; RuntimeLoop rejects the decision kind and has no state-patch branch. | **Moderate** | The mutation service already exists, but RuntimeLoop would need an explicit decision kind, validation branch, event/result contract, and tests before Runtime removal. |
| Decision retry / parse-failure loop | Runtime can retry invalid model decisions, tool-intent failures, and strict JSON parse failures with specialized retry context; RuntimeLoop performs one provider call and returns rejected/provider-failed outcomes. | **Significant** | Porting this changes provider interaction semantics, event vocabulary, context shape, and failure contracts. |
| Tool-intent guard | Runtime rejects tool calls that do not match user intent before execution and can retry; RuntimeLoop relies on the provider/builder and shared tool validation but does not enforce this behavior. | **Moderate** | The guard exists, but RuntimeLoop would need to decide how to apply it against `RuntimeContext.tools`, how to journal failures, and whether to retry or simply reject. |
| Distinct question/refusal response kinds | Runtime exposes `question` and `refusal`; RuntimeLoop intentionally emits answer-shaped clarify/refuse responses and rejects direct `ask_question` / `refuse` decisions. | **Trivial** if the answer-mapping is accepted as the target; **moderate** if distinct RuntimeLoop decision/result kinds must be restored. | The main unresolved issue is compatibility expectation, not missing low-level infrastructure. |
| Legacy context packet vs RuntimeLoop context view | Runtime providers receive budgeted `ContextPacket` data and retry prompts; RuntimeLoop providers receive `RuntimeContext` plus `DecisionContextView`. | **Significant** if legacy provider compatibility is required; **trivial** if legacy providers can be retired. | The effort depends on whether old prompt/model clients must keep working unchanged. |
| Registered operation import-path execution vs supplied handlers | Runtime executes registered operation import paths through `ToolExecutor`; RuntimeLoop only executes explicitly injected handlers. | **Moderate** | Tool validation/policy is shared, but execution dispatch semantics and failure events differ. |
| Legacy CLI `--plan` path | `--plan` still calls `run_legacy`; default CLI, shell without `--plan`, HTTP, and API paths use RuntimeLoop. | **Trivial** after the needed Runtime-only behaviors are either ported or intentionally dropped. | This is a routing dependency, but changing it before semantic decisions are made could remove user-visible behavior. |

## Could Runtime be removed today?

**Not yet.**

RuntimeLoop is now the primary path for the API, default CLI, ordinary shell messages, answer/tool-request/tool-call flows, shared tool recommendation, shared tool validation, shared validation-before-policy, RuntimeLoop provider-failed handling, decision journaling, and traceability. However, deleting `Runtime` today would still remove several behavior-level capabilities that are present in current source:

- model-proposed state mutation through `propose_state_patch` and `StatePatchService`;
- pending confirmation/approval creation and approved pending-action resume through `ToolExecutor`;
- retry and parse-failure correction behavior for model decisions;
- tool-intent rejection before tool execution;
- distinct legacy question/refusal response contracts; and
- the CLI `--plan` path's current `run_legacy` behavior.

The answer is **Not yet**, rather than **No**, because the default application path already uses RuntimeLoop and the major shared infrastructure gaps have been extracted. It is also not **Almost**, because at least two behavior-level gaps remain substantial enough that Runtime removal would still be a visible semantic regression.

## Highest leverage next target

**Pending confirmation/approval and approved-action resume** is the highest leverage next target.

This is the best single target because it is the largest remaining user-facing safety and workflow gap. The recommendation, validation, and validation-before-policy pieces are already shared, so the remaining work is now concentrated in the behavior RuntimeLoop applies after a policy outcome is returned. Closing this gap would align RuntimeLoop with Seed's policy model for `require_confirmation` and `require_approval`, preserve human-gated execution semantics, and reduce the biggest blocker to making RuntimeLoop the only runtime path. State patches are important, but their mutation service already exists and their route is narrower; pending-action parity affects every policy-gated tool call and therefore has higher architectural leverage.
