## Suspect ownership mismatches

| Misleading implication | Actual boundary |
|---|---|
| "Runtime owns pending actions." | `PendingActionService` owns creation/status events; `ToolExecutor` owns policy-denied pending-action routing and approved resume. Runtime only invokes execution. |
| "Runtime owns state patches." | `StatePatchService` translates patch ops into ledger events; Runtime only routes model decisions and records rejection. |
| "RuntimeLoop owns recommendations." | `ToolRecommendationService` and `RecommendationRanker` own ranking; RuntimeLoop only calls them after inline tool-need creation. |
| "CLI owns observations." | `ObservationIngestor` owns observation/evidence/fact emission; CLI parses input and delegates. |
| "StateProjector owns facts/evidence creation." | Projector only reads events into state; creation belongs to ingestion, state patching, fact extraction, and registration/action services. |
| "API owns runtime behavior." | `SeedAPI` is a dependency-light adapter over canonical Runtime, StateProjector, and ToolRegistry; RuntimeLoop adapter references are historical/quarantined. |
| "RuntimeLoop uses the full ToolExecutor path." | RuntimeLoop uses `ToolExecutionPolicyService` but dispatches to its own in-memory handler map and emits `tool.result`; `ToolExecutor` owns registered-function dispatch and pending-action lifecycle. |
| "Context composition is one service." | Legacy Runtime uses `DecisionInputComposer`; RuntimeLoop composes `RuntimeContext` inline plus `DecisionContextView`. |
| "Decision validation is centralized." | Legacy Runtime uses `DecisionValidator`; RuntimeLoop has its own `_validate_decision` shape validation. |
| "Operation registration is only registry mutation." | `ToolRegistry` owns in-memory lookup; `ToolkitRegistrationService` also emits `tool.registered` so `StateProjector` can expose registered operations in state. |

## Immediate architectural conclusions

1. **Clean reusable services today:** `StateProjector`, `ProjectionStore` helpers, `ToolValidationService`, `ToolExecutionPolicyService`, `PolicyGate`, `ToolRecommendationService`/`RecommendationRanker`, `FactExtractionService`, `PendingActionService`, `StatePatchService`, `ObservationIngestor`, `DecisionJournal`, `RuntimeTraceReader`, state/evidence/confidence/context view builders, and action/handoff/proposal services.
2. **Owners that are too fat or mixed:** `RuntimeLoop` mixes orchestration with user-message ingestion, decision validation, request-tool creation, in-memory tool dispatch, and runtime-specific event emission; `Runtime` mixes orchestration with retry prompts, intent guarding, and state-patch routing; `scripts/seed_local.py` mixes CLI routing, app wiring, local HTTP, dev seeding, formatting, and planning helpers.
3. **Caller-specific capabilities that should stay caller-specific:** CLI routing/formatting/dev seeding, API adapter methods, Runtime/RuntimeLoop orchestration policy, RuntimeLoop in-memory test handler dispatch, and runtime trace/why presentation.
4. **Runtime gaps that are actually caller wiring:** API/CLI exposure for projected state, trace/why, operation/provider listing, recommendations after ToolNeeds / capability gaps, projection-cache usage, observation ingestion commands, and action/handoff plan commands mostly wire existing services.
5. **Runtime gaps that require real semantic design:** shared user-message ingestion/run identity, one canonical decision-validation contract, unified context composition for Runtime and RuntimeLoop, whether RuntimeLoop should use `ToolExecutor` or keep separate handler dispatch, first-class pending-action approval/resume routes, and explicit ownership for request-tool creation in RuntimeLoop versus `ToolNeedService`.
