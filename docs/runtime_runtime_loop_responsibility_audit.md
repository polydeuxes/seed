# Runtime and RuntimeLoop Responsibility Duplication Audit

> **Stale/quarantined RuntimeLoop-era inventory.** Current architecture treats `Runtime` as canonical and `RuntimeLoop` as deprecated/experimental, not CLI/API/default behavior. References to legacy/old Runtime or RuntimeLoop migration are historical audit wording, not current-core guidance; current architecture treats Runtime as canonical.


Scope: file-level inventory only. This audit does not propose behavior changes, does not refactor runtime code, and does not delete `Runtime` or `RuntimeLoop`.

## Cross-cutting findings

1. **The highest-risk duplication is not validation logic; it is orchestration result semantics.** `ToolExecutionPolicyService`, `ToolValidationService`, `PendingActionService`, `StatePatchService`, `DecisionJournal`, and `ProjectionStore` already provide shared service boundaries. The remaining duplicated responsibilities are mostly the runtime-specific glue that turns service results into events, journal records, and public response/result shapes.
2. **RuntimeLoop has already extracted request-tool handling, decision validation, and context composition into separate files.** These are good precedents for future extraction, but they intentionally preserve RuntimeLoop behavior rather than becoming shared Runtime/RuntimeLoop abstractions.
3. **Pending actions are the biggest parity gap.** Runtime creates and resumes pending actions; RuntimeLoop denies all non-allow policy outcomes. Treat this as behavior design work, not a mechanical deduplication.
4. **State patching is a canonical Runtime-only route today.** The canonical mutation service exists, but RuntimeLoop has no state-patch decision contract or journal/result vocabulary for it.
5. **Event vocabulary must be treated as API.** Runtime and RuntimeLoop tests assert different event kinds and ordering. Any extraction must preserve event emission exactly or include a deliberate migration plan.

## Recommended no-refactor sequence

1. Keep `Runtime` and `RuntimeLoop` intact.
2. If future changes are approved, extract RuntimeLoop-only outcome recording first because it is a low-behavior-risk move that reduces method size without crossing semantic boundaries.
3. Extract shared pure helpers next: visible-tool projection, request-tool payload-to-`ToolNeed` construction, and decision field primitive checks.
4. Defer shared execution/pending-action adapters until RuntimeLoop's `require_confirmation`/`require_approval` semantics are explicitly specified and tested.
