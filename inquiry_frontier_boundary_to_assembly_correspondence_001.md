# Inquiry frontier boundary testimony to bounded frontier assembly correspondence 001

## Scope

Inspected only the immediate handoff:

```text
InquiryFrontierBoundaryTestimony
→ bounded inquiry frontier assembly
```

The comparison tested this distinction:

```text
inquiry-frontier boundary testimony
!=
inquiry frontier establishment
```

## Recovered correspondence

| Required recovery | Finding |
| --- | --- |
| standing entering the handoff | `InquiryFrontierBoundaryTestimony` is read-only preserved boundary testimony for one exact selected inquiry need. Its own boundary notes say it does not assemble a frontier, formulate a question, open inquiry, authorize access, execute, record, write the event ledger, or mutate cluster state. |
| identity actually validated | `assemble_bounded_inquiry_frontier(...)` validates selection/testimony identity across selected reference id, native projection id, native lineage, need set id, selected need selection id, selected goal establishment id, and horizon id. Identity mismatch becomes `identity:selection-testimony` material conflict. |
| lineage merely preserved | Source testimony, bounded uncertainty component, repository/world subject, native lineage, testimony id, and all preserved clause references are carried into `BoundedInquiryFrontier`; preserved lineage is not itself validated lineage beyond the explicit selected-need/testimony identity checks. |
| additional material required | Establishment requires operative coherent clauses for all four required families: included/excluded inquiry scope, eligible/ineligible evidence territory, sufficient-resolution conditions, and lawful stopping conditions. Unsupported, unknown, mixed, adjacent-family, stale, unavailable, out-of-scope, or conflicting clauses may be preserved, but do not supply operative establishment. |
| standing established by assembly | When the selected need is an inquiry need, identity is coherent, no material binding conflict exists, and all four required clause families are operatively coherent, assembly returns `frontier_state == "established"`. |
| standing explicitly not established | Boundary testimony does not establish a frontier by itself. Frontier establishment is also explicitly not question formulation, inquiry opening, access authorization, examination/execution start, recording, event-ledger write, cluster mutation, source selection, observation selection, or result knowledge. |

## Answer

Frontier assembly does not merely consume preserved boundary testimony as already-established frontier standing. It consumes the preserved testimony, validates the selected-need/testimony identity, checks operative clause coherence and required-family completeness, preserves non-operative lineage separately, and only then establishes bounded frontier standing.

The assembly faithfully consumes boundary testimony
and establishes bounded frontier standing
without opening inquiry.
