## Independently recurring implementation movements

### 1. Compressed builder or composer responsibility moved into named local producers

All three families begin from a public-facing builder/composer that had more responsibility than the adjacent implementation evidence required, then recover narrower local producers:

- Inquiry Orientation identifies `_compose_inquiry_orientation_answer(...)` as the compressed producer for bounded profile composition, then completion evidence shows the separable producers already own request preparation, evidence collection, selected-material preservation, reason selection, authority-boundary selection, limitation selection, and answer preparation.
- Reasoning Path starts with `build_reasoning_path_audit(...)` compressing collection, selection, evidence projection, conclusion derivation, consumer preservation, story impact, typed Unknowns, payload assembly, and compatibility handoff; each slice recovers one adjacent producer from that body.
- Selection Path starts with `build_selection_path_audit(...)` compressing root preparation, target normalization, input gathering, implemented target resolution, branch production, unsupported production, and compatibility handoff; slices recover target resolution, branch result production, payload bundle production, unsupported bundle production, and compatibility construction.

This is constitutional recurrence because each family independently records a builder/composer body being narrowed by implementation evidence into local producers, not because a preferred style favors helper extraction.

### 2. Data handoff became explicit through implementation-local artifacts or existing payloads

All three families make ownership inspectable by carrying handoffs through local artifacts or payloads:

- Inquiry Orientation records `_InquiryOrientationCompositionRequest`, `_InquiryOrientationEvidence`, `_InquiryOrientationSelectedMaterial`, `_InquiryOrientationAnswer`, and `InquiryOrientationView` as adjacent artifacts that carry the local handoffs.
- Reasoning Path records `_DerivationSupportingEvidencePayload`, `_DerivedConclusionPayload`, `_DerivationLineagePayload`, typed Unknown records, and `ReasoningPathAudit` as sufficient terminal artifacts.
- Selection Path records `_SelectionTargetSelection`, `_SelectionPathPayloads`, `_SelectionLineagePayload`, member payloads, typed Unknown payloads, and `SelectionPathAudit` as the artifacts that carry recovered boundaries or already-lawful ownership.

This is constitutional recurrence only at the level of explicit local handoff carriers. It does not support a universal artifact ontology, registry, or profile framework.

### 3. Public compatibility and read-only boundaries were preserved while local ownership changed

All three families independently preserve public compatibility while recovering implementation-local ownership:

- Inquiry Orientation preserves CLI behavior, rendering behavior, note storage behavior, deterministic lexical overlap, negative authority, no projection into runtime state, no event-ledger writes, and no cluster mutation.
- Reasoning Path slices repeatedly preserve public CLI behavior, JSON/text rendering, `ReasoningPathAudit`, read-only behavior, evidence/conclusion/lineage/story/Unknown preservation, no event-ledger writes, no fact recording, and no cluster mutation.
- Selection Path slices preserve CLI behavior, JSON/text rendering, `SelectionPathAudit`, existing payload/helper artifacts, read-only behavior, typed Unknown behavior, lineage, candidate, evidence, no fact recording, no event-ledger writes, and no cluster mutation.

This recurrence is constitutional because the same compatibility-preserving movement appears in all three independent families while implementation-local ownership changes are admitted only behind those boundaries.

### 4. Recovery proceeded one implementation-local boundary at a time

All three families support one-boundary-at-a-time movement:

- Inquiry Orientation pressure and completion evidence isolates one bounded profile-composition neighborhood and then stops when only orchestration remains.
- Reasoning Path slices each name exactly one implementation-local ownership boundary, including supporting evidence, relevant-row selection, conclusions, consumers, fallback, story impact, and typed Unknowns.
- Selection Path slices each name a bounded next ownership boundary, and the payload ownership audit refuses to count already visible payload-member ownership as new recoverable slices.

This is constitutional recurrence because all three independent progressions use repository evidence to admit only a local movement and refuse broader redesign.

### 5. Remaining orchestration was not treated as ownership pressure

All three families independently end by distinguishing terminal sequencing from recoverable ownership:

- Inquiry Orientation completion says `_compose_inquiry_orientation_answer(...)` now only sequences evidence collection, selected-material preparation, and answer preparation.
- Reasoning Path completion says remaining work is upstream surface collection, sequencing recovered producers, terminal accumulation, payload assembly, and compatibility handoff.
- Selection Path completion says remaining visible helpers and public fields are already bounded and that another slice would merely relocate already-owned implementation.

This recurrence is constitutional because all three families stop at the same kind of implementation condition: shortening or wrapping orchestration is not a lawful recovery if no new owner, artifact, or helper responsibility is exposed.

## Independently recurring ownership transitions

### 1. Public builder/composer retained orchestration while narrower producers received local ownership

Across all three families, ownership did not move into a global mechanism. It moved from an over-compressed public-facing builder/composer into narrower local producers, while the builder/composer retained lawful sequencing and adaptation:

- Inquiry Orientation leaves `_compose_inquiry_orientation_answer(...)` as an inquiry-local orchestrator after evidence, selected material, and answer construction become owned by narrower producers.
- Reasoning Path leaves `build_reasoning_path_audit(...)` as terminal collector/sequencer/payload assembler after derivation-local producers are recovered.
- Selection Path leaves `build_selection_path_audit(...)` and branch handoffs as dispatch/sequencing/adaptation points after target, branch, payload, lineage, and compatibility responsibilities become explicit.

### 2. Artifacts carried bounded responsibility; consumers did not backfill producer ownership

Across all three families, once a producer/artifact boundary is visible, downstream consumers use the carried material without becoming owners of producer-side computation:

- Inquiry Orientation has `build_inquiry_orientation(...)` consume `_InquiryOrientationAnswer` to build `InquiryOrientationView`; rendering consumes the view, not evidence internals.
- Reasoning Path has `build_reasoning_path_audit(...)` consume local producer outputs into payloads and `_reasoning_path_from_payloads(...)` project payloads into `ReasoningPathAudit` without owning derivation production.
- Selection Path has `_selection_path_from_payload_bundle(...)` consume `_SelectionPathPayloads` and project public fields without selecting candidates, creating factors, gathering evidence, producing lineage, preserving Unknowns, recording facts, writing the event ledger, or mutating cluster state.

### 3. Unknown and negative-authority boundaries remained bounded rather than becoming truth

All three families preserve Unknown or negative-authority boundaries as bounded output, not as cluster truth:

- Inquiry Orientation preserves uncertainty and negative authority around deterministic lexical overlap and refuses semantic interpretation or note promotion.
- Reasoning Path recovers typed Unknown preservation for evidence gaps while preserving no fact recording, no event-ledger writes, and no cluster mutation.
- Selection Path preserves unsupported targets and typed Unknown payloads as selection-lineage/public audit material, with the payload ownership audit confirming typed Unknown production, transport, and public conversion are already separately owned.

## Independently recurring completion conditions

The implementation completion condition independently recurring across all three families is:

```text
Implementation recovery is complete when repository evidence no longer exposes an adjacent, implementation-local, recoverably compressed ownership boundary, and the remaining code is lawful orchestration, payload/public compatibility handoff, upstream surface collection, rendering/view adaptation, or already-owned payload/helper responsibility.
```

Family-specific support:

- Inquiry Orientation: no additional producer, artifact, helper, or consumer is exposed; another slice would only wrap the existing call chain.
- Reasoning Path: no remaining derivation producer, evidence projection owner, consumer-preservation owner, typed Unknown owner, or public-shape artifact is exposed; another slice would move collection, sequencing, payload construction, or compatibility handoff.
- Selection Path: no adjacent producer or helper is both unowned and recoverably compressed; another slice would relocate already-owned implementation, and payload-member responsibilities have been individually classified as already lawfully owned.

## Recurring distinctions supported by all three families

### Implementation ownership is not public compatibility ownership

All three families separate private/local implementation recovery from public compatibility objects:

- Inquiry Orientation distinguishes `_InquiryOrientationAnswer` from `InquiryOrientationView` and rendering.
- Reasoning Path distinguishes derivation payloads and local producers from `ReasoningPathAudit`.
- Selection Path distinguishes target/payload/lineage/member producers from `SelectionPathAudit`.

### Visibility is not ownership pressure

All three families show visible fields, helpers, or public surfaces do not by themselves justify another recovery:

- Inquiry Orientation visible orchestration is not enough once all producer/artifact handoffs are explicit.
- Reasoning Path visible upstream calls and payload assembly are not enough once derivation work is owned.
- Selection Path visible public fields and many helpers are not enough once payload and lineage responsibilities are already lawful.

### Orientation/audit output is not cluster mutation

All three families preserve read-only boundaries and refuse to treat diagnostic/orientation/audit output as fact recording, event-ledger write, or cluster mutation.

## Constitutional recurrence vs implementation coincidence

The following observations are constitutional recurrence because all three independent families support them:

1. Recovery is evidence-selected and local, not preference-selected or framework-selected.
2. Each recovery movement admits only one adjacent implementation-local ownership boundary at a time.
3. Local producers/artifacts/helpers make ownership inspectable before public compatibility construction.
4. Public compatibility is preserved while private ownership changes.
5. Consumers use carried payload/profile/view material without backfilling producer authority.
6. Unknowns, limitations, and negative authority remain bounded and non-mutating.
7. Completion occurs when remaining movement would wrap or relocate orchestration rather than recover an unowned local responsibility.

The following are implementation coincidences or family-local facts, not constitutional grammar:

- Specific helper names such as `_reasoning_path_typed_unknowns(...)`, `_selection_path_from_payload_bundle(...)`, or `_prepare_inquiry_orientation_answer(...)`.
- The number of slices in a campaign.
- The exact payload field names in any one family.
- Whether a new artifact was required in a given slice; Reasoning Path Slice 009 required no new payload artifact, while other slices used existing or new local carriers.
- The concrete domain vocabulary of inquiry notes, derivation paths, target selection, pressure categories, or candidate rows.

## Can future implementation recovery campaigns rely upon the recovered grammar?

Yes, but only as a repository-evidence constraint, not as a planner, workflow, engine, or mandate.

A future implementation recovery campaign may rely on the recovered grammar to ask whether repository evidence exposes:

1. an adjacent implementation-local compressed responsibility;
2. a narrower lawful producer, artifact, helper, or consumer boundary;
3. preserved public compatibility and read-only/mutation boundaries;
4. Unknown, limitation, lineage, support, or negative-authority material that must remain bounded;
5. a completion point where further movement would only wrap orchestration or relocate already-owned implementation.

A future campaign may not rely on this survey to infer a slice without implementation evidence, require a particular helper shape, require a payload bundle, introduce a framework, or continue recovery after the local completion condition is met.
