# Seed Constitution

## Status

This document is a repository orientation artifact. It preserves the governing discipline recovered across Seed's architectural recovery, null-state transition, pressure visibility, inquiry, completion-audit, answer-composition, provider-translation, capability-verification, diagnostic-visibility, question-dispatch, repository-orientation, and operator handoff work.

It is constitutional only in the narrow sense that it records rules for legitimate knowledge transition and implementation change inside this repository. It is not runtime behavior. It does not create a new implementation family.

If this document conflicts with implementation evidence, implementation evidence wins.

## Purpose of this document

This document answers:

```text
What rules govern legitimate knowledge transition,
architectural recovery,
inquiry,
pressure visibility,
artifact handoff,
and implementation change
inside Seed?
```

It helps future sessions, operators, and implementation workers unload local context and reorient quickly without promoting local reasoning into repository truth.

The central discipline is:

```text
Null / unknown / uncommitted
-> observation
-> bounded unknown
-> inquiry
-> evidence
-> supported transition or explicit stop
-> bounded handoff artifact
```

## Constitutional rules

### 1. Null persists until evidence justifies transition

The default state is not belief or disbelief. The default state is:

```text
Unknown / uncommitted
```

A subject remains Null until observable evidence justifies movement. This governs claims, facts, observations, ownership, pressure, inquiry, classification, authority, compatibility, and recovery.

No competency bypasses Null. A note, provider response, report, audit row, code corridor, compatibility residue, or operator concern may orient work, but it does not become repository truth by appearing.

### 2. Observation precedes inquiry

Inquiry does not begin from global purpose, planner intent, architectural preference, or a desire to complete a framework.

Inquiry begins when an observation creates a bounded unknown. The observation may be external or internal:

```text
an unknown repository
a provider response
a report
a pressure audit row
a code corridor
a compatibility residue
.
```

Observation creates orientation. It does not create truth.

### 3. Evidence moves Null

Only evidence may move a subject from Null toward a supported state. Evidence may support states such as:

```text
candidate
supported
bounded
verified
trusted
```

Unsupported evidence must leave the subject uncommitted. "Insufficient implementation evidence" is a valid successful result, not a rhetorical failure.

### 4. Repository authority wins

Repository authority does not mean the code is always conceptually correct. It means repository-visible implementation, tests, reports, diagnostics, observations, and compatibility behavior are the court of evidence.

Architectural preference, repeated vocabulary, operator intuition, or attractive abstractions cannot overrule repository evidence.

If this document conflicts with implementation evidence, implementation evidence wins.

### 5. No owner without recurring implementation evidence

A responsibility owner is presumed not to exist until recurring implementation evidence demonstrates it.

Do not create ownership from:

- appealing concepts;
- repeated vocabulary alone;
- presentation prose;
- operator intuition;
- future aspiration;
- one-off coincidence.

Ownership recovery requires implementation-backed compression: behaviorally real bounded responsibilities already carried by an owner or corridor, with enough evidence to recover a boundary or handoff.

### 6. Recovery relieves implementation pressure

Pressure exists when one owner or corridor carries multiple behaviorally real bounded responsibilities without enough independent boundary, authority, compatibility, or stopping visibility.

Recovery does not search for abstractions. Recovery searches for compressed bounded work.

A successful recovery slice relieves one compatibility-preserving pressure by making one local boundary, owner, result, request, payload, helper, or handoff visible enough to reason about, preserve, and test.

### 7. Pressure informs inquiry; it does not command it

Pressure identifies what deserves attention. It does not automatically select work, authorize implementation, create purpose, recover ownership, or promote a frontier.

Pressure is an instrument, not the north star. A pressure row, pressure report, operator concern, or residual pressure item is orientation evidence until repository evidence supports a bounded inquiry or implementation change.

### 8. Inquiry is bounded work

Inquiry answers one bounded question. It should preserve:

```text
observation
evidence
classification / conclusion
authority
stop condition
```

Recovery, pressure visibility, capability verification, answer composition, repository orientation, question-family dispatch, diagnostic shape audit, and completion audit are specialized inquiry forms when they follow this discipline.

### 9. Authority must be explicit

Every surface must know what it may and may not do.

The following distinctions must be preserved:

```text
Visibility is not authority.
Diagnostic output is not cluster truth.
Projection is not verification.
Observation is not fact.
Candidate is not verified.
Pressure is not recovery.
Answer is not mutation.
Frontier is not implementation readiness.
```

Read-only surfaces must remain read-only. Diagnostic findings must not silently become cluster truth. Presentation vocabulary must not become repository knowledge without implementation evidence.

### 10. Compatibility preservation is the default recovery constraint

Implementation slices should preserve public behavior unless a compatibility break is explicitly justified.

Most recovered boundaries have been implementation-local handoffs, payloads, candidates, requests, results, or helpers that preserve public JSON, CLI, rendering, event, ledger, cache, diagnostic, and projection behavior.

Compatibility preservation is not conservatism for its own sake. It is the condition that lets Seed recover architecture without changing what downstream users or tools already rely on.

### 11. Stop is a positive result

Stopping is not failure. Valid stops include:

```text
insufficient implementation evidence
unsupported conclusion
unknown target
adjacent-family pressure
family complete
methodology only
frontier only
not implementation-backed
```

A system that cannot stop cannot be trusted. Stop conditions preserve authority boundaries and prevent unsupported transition from Null into ownership, truth, implementation, or recovery.

### 12. Competencies exchange artifacts, not shared internal state

Each competency performs bounded local work and exports an artifact. Artifacts, not hidden reasoning, cross boundaries.

Examples:

```text
Observation -> Evidence
Evidence -> Claim Support
Slice Report -> Handoff Prompt
Pressure Audit -> Pressure Item
Candidate -> Public Compatibility Object
Codex Report -> Next Inquiry Prompt
```

Competencies may reason locally, but they must hand off bounded artifacts.

### 13. Local reasoning, bounded handoff

A competency, operator, assistant, Codex session, or implementation worker may have rich local state. That local state should not leak as authority.

Only the bounded handoff artifact is shared. This allows Codex, assistant, operator, and Seed competencies to cooperate without requiring one monolithic mind.

A handoff should make its observation, evidence, conclusion, authority, and stop condition visible enough that the next worker can continue without inheriting unbounded private context.

### 14. Frontiers preserve pressure that recovery cannot yet relieve

A frontier is not a roadmap item. A frontier preserves important pressure that is not yet implementation-recoverable.

Recovery stops when evidence is insufficient, when same-family compression is exhausted, or when remaining pressure changes ownership family. Frontier work preserves unanswered pressure without pretending implementation authority exists.

A frontier does not create a planner, priority, schedule, runtime behavior, or implementation mandate.

### 15. Completion audits terminate families

Completion audits are gates. They determine whether same-family pressure remains.

They may conclude:

```text
complete
continue
reassign to another family
frontier
insufficient evidence
```

Completion audits prevent endless decomposition. Completion means no supported same-family compressed owner remains; it does not mean all adjacent pressure has disappeared.

### 16. The first observation is enough to begin

Seed does not need a full map to begin. Seed needs one observation.

From that observation:

```text
Observation
↓
Bounded unknown
↓
Inquiry
↓
Evidence
↓
Supported transition
↓
Next observation
```

This is the smallest self-sustaining inquiry loop. It is enough to orient without inventing a total architecture, goal system, ontology, planner, or map.

## What this document is not

This document is not:

- runtime code;
- an ontology;
- a schema;
- a roadmap;
- a planner;
- a goal system;
- a motivation engine;
- an agent architecture;
- automatic recovery authority;
- a replacement for repository evidence;
- a design spec;
- a test plan;
- a diagnostic surface;
- a source of cluster truth.

It records recovered governing discipline. It does not implement that discipline.

## How to use this document

Use this document as an orientation checklist before changing Seed or interpreting Seed output:

1. Identify the observation.
2. State the bounded unknown.
3. Gather repository-visible evidence.
4. Classify the supported transition, if any.
5. Name the authority boundary.
6. Preserve the stop condition.
7. Export a bounded artifact for the next worker.

If any step lacks evidence, stop at Null, candidate, unsupported, frontier, or insufficient implementation evidence as appropriate.

## Relationship to implementation

Implementation change requires implementation evidence. This document may help frame a change, but it cannot authorize one.

A legitimate implementation recovery slice should normally:

- start from an observed pressure or bounded unknown;
- identify a behaviorally real compressed responsibility in implementation;
- recover one local boundary or handoff;
- preserve public compatibility unless a break is explicitly justified;
- prove preservation with existing repository conventions;
- stop after one supported transition unless new same-family evidence remains.

Documentation vocabulary does not become preserved or projected knowledge unless implementation evidence supports that transition.

## Relationship to frontiers

Frontiers preserve important unanswered pressure. They do not rank work, schedule work, select slices, prove implementation readiness, or imply that recovery failed.

A frontier is appropriate when pressure is real enough to preserve but not yet recoverable as implementation-backed work.

## Relationship to recovery

Recovery is compatibility-preserving relief of implementation pressure. It is not abstraction hunting, architecture beautification, or framework construction.

Recovery begins when evidence shows one owner or corridor carrying multiple bounded responsibilities whose boundary, authority, compatibility, or stopping point is not independently visible enough. Recovery stops when the selected family no longer has supported same-family compression.

## Relationship to operators and tools

Operators and tools may provide observations, pressure, reports, probes, commands, and handoff artifacts. Those inputs orient inquiry, but they do not automatically become truth, authority, ownership, or work selection.

Codex, assistants, operators, diagnostic surfaces, and Seed competencies should exchange bounded artifacts rather than shared hidden state. Each handoff should make enough evidence and authority visible for the next step to proceed safely or stop.

## Open boundaries

This document intentionally leaves these boundaries open:

- which future frontiers become implementation-backed;
- whether a currently methodological competency later gains recurring implementation evidence;
- whether a vocabulary term deserves preservation as repository knowledge;
- whether a pressure item is same-family, adjacent, residual, unsupported, or frontier-only;
- whether a compatibility break is justified in a future task.

Each open boundary must be resolved by repository evidence at the time of work. The constitutional rule remains: Null persists until evidence justifies transition.
