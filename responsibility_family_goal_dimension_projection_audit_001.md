# Responsibility Family to Goal Dimension Projection Audit 001

## Scope

This is an audit-local repository investigation. It does not add a diagnostic, CLI flag, runtime registry, static goal-dimension registry, planner, scheduler, prioritizer, authorization surface, execution surface, inquiry transition, goal inventory, pressure association, event-ledger writer, or cluster mutation.

The bounded question is whether implementation-backed evidence permits one repository responsibility family to be projected as an orthogonal goal dimension, and whether that projection can be derived read-only from existing responsibility ownership, artifacts, diagnostics, and preserved boundaries.

Repository authority wins. Presentation vocabulary is not treated as knowledge unless implementation evidence or repository-artifact authority supports it.

## Guardrail preservation

The audit preserves these boundaries:

```text
responsibility family
!= goal dimension automatically

goal dimension
!= pressure

pressure
!= bounded goal

bounded goal
!= inquiry authority
!= execution authority

derived projection
!= runtime registry
!= planner
!= priority order
!= universal goal ontology
```

A lawful projection can only say that a responsibility family is stable enough to be named as one possible orthogonal dimension for future reasoning. It cannot say the dimension has pressure, that a goal exists, that an inquiry should move, or that work is authorized.

## Lawful boundary

The lawful boundary is:

```text
ResponsibilityFamilyEvidence
-> GoalDimensionProjection
```

That boundary is read-only and derivational. It means a projected dimension is supported only when repository evidence already preserves a stable responsibility-family identity, its ownership boundary, its artifact or implementation evidence, its consumers, and its refusals.

The boundary is not:

```text
ResponsibilityFamilyName
-> GoalDimensionProjection
```

A name alone is insufficient. A slice title, presentation label, or answer-responsibility string can orient investigation, but it does not prove a goal dimension without implementation-backed or repository-authority evidence.

The boundary is also not:

```text
GoalDimensionProjection
-> pressure
-> bounded goal
-> inquiry movement
-> execution authority
```

The projection stops before pressure. A dimension may remain Null.

## Evidence sufficient to prove projection support

A responsibility family is stable enough to support a goal-dimension projection when the evidence set contains all of the following:

1. **Stable family identity.** The family is not merely a prose label; it recurs across implementation slices, repository-artifact authority, or tested runtime surfaces.
2. **Ownership boundary.** The family has an explicit statement of what it owns and what it refuses.
3. **Implementation-backed or repository-authority support.** For implementation families, compatibility-preserving code and tests or generated architecture evidence support the boundary. For repository-artifact families, scoped documentation authority supports the boundary and explicitly refuses runtime ownership.
4. **Consumer evidence.** At least one existing surface, report, artifact, or workflow consumes the family boundary without transferring ownership.
5. **Remaining-compression accounting.** The evidence distinguishes completed or stable boundaries from adjacent compressed boundaries and does not overclaim exhaustion.
6. **Boundary refusals.** The evidence preserves what the family is not authorized to do, especially pressure creation, goal activation, inquiry movement, planning, scheduling, authorization, execution, event-ledger mutation, and cluster mutation.
7. **Derivability without stored goal state.** The projection can be recomputed from existing evidence rather than stored in a static goal-dimension registry.

If any of these elements is absent, the projection remains Unknown or unsupported.

## Unsupported or Unknown candidates

The following must remain Unknown or unsupported:

- **Universal goal ontology.** Repository evidence explicitly supports local families and refusals, not a total goal system.
- **Static goal-dimension registry.** No implementation-backed owner currently requires stored dimension declarations.
- **Goal inventory or goal-state inventory implementation.** A prior topology audit found the responsibility plausible but not implementation-ready without lawful source records.
- **Pressure association.** A supported dimension does not prove any pressure exists for that dimension.
- **Bounded-goal creation.** A dimension does not create a `Goal`, provisional operator goal, or event-ledger record.
- **Inquiry movement.** No projection authorizes inquiry selection, inquiry transition, or current-focus movement.
- **Prioritization or scheduling.** Existing context ordering and pressure selection are surface-local and do not authorize a cross-family rank.
- **Authorization or execution.** Operational responsibility boundaries explicitly separate recommendation, validation, policy authorization, realization, recording, and extraction; a dimension projection cannot skip those boundaries.
- **Presentation-only vocabulary.** Terms that appear only as labels remain unsupported until implementation evidence or repository-artifact authority preserves them.

## Provenance and boundary refusals

The projection should preserve provenance and boundary refusals if it is ever implemented or reused beyond this audit.

Minimum provenance fields would be:

```text
family_id
projected_dimension_label
support_level
evidence_refs
owner_boundary
consumer_refs
known_refusals
remaining_unknowns
derivation_timestamp_or_source_revision
```

The most important fields are `evidence_refs`, `known_refusals`, and `remaining_unknowns`. Without those fields, a projection artifact would be prone to becoming an ungrounded registry, planner input, priority list, or implicit authorization surface.

Boundary refusals are not decoration. They are part of the evidence that makes projection lawful.

## Does an existing owner already perform this derivation?

No existing owner appears to perform the full derivation from responsibility-family evidence to goal-dimension projection.

Closest existing owners are narrower:

- The completion audit manually composes family status from slice evidence, but it is not a runtime owner and does not project goal dimensions.
- The orthogonal goal inventory topology audit describes a possible read-only inventory need, but it does not identify lawful source records or implement a projection owner.
- Question surface inventory owns question-family answer responsibilities, not cross-family goal dimensions.
- Pressure audit owns pressure visibility and ordering for pressure items, not family-to-dimension projection.
- Context selection orders active projected goals, not Null dimensions or responsibility families.
- Repository-artifact ownership slices recover documentation authority, not runtime projection derivation.

Therefore the derivation currently exists only as audit-local reasoning over preserved evidence.

## Is one shared projection artifact warranted?

One shared projection artifact is warranted only as a future read-only derivation artifact if another bounded question needs repeated access to this mapping.

A shared artifact would be justified if future work repeatedly asks:

```text
Which responsibility families are stable enough to be considered orthogonal goal dimensions, and what refusals keep those dimensions from becoming pressure, goals, inquiries, priorities, or execution?
```

Even then, it should be derived, not manually curated as a static registry. The safer shape is a generated or audit-local evidence projection that can be recomputed from family evidence and repository-artifact authority.

A static registry is not warranted now because:

- it would invite ontology drift;
- it would obscure provenance;
- it would risk converting dimensions into backlog categories;
- it would duplicate authority already preserved in slices, audits, tests, and documentation-authority artifacts;
- no current implementation consumer requires it.

## Is implementation necessary now?

Implementation is not necessary now.

Audit-local preservation is sufficient because the deliverable is an investigation report, no operational command is added, no diagnostic or recordable output changes, and no consumer currently needs a machine-readable projection.

If implementation is later requested, the smallest lawful implementation would be a read-only derivation surface that:

- reads existing family evidence;
- emits provenance and boundary refusals;
- preserves Unknowns;
- refuses pressure, goal creation, inquiry movement, priority, authorization, and execution;
- is added to diagnostic inventory and diagnostic shape audit if exposed as a diagnostic or operational visibility surface;
- includes tests proving visibility and refusal boundaries.

This audit does not implement that surface.

## Determination

Implementation-backed evidence permits a repository responsibility family to be projected as an orthogonal goal dimension only when the family has stable identity, explicit ownership, implementation-backed or repository-authority support, consumer evidence, remaining-compression accounting, and preserved refusals.

The lawful boundary is:

```text
ResponsibilityFamilyEvidence
-> GoalDimensionProjection
```

The projection can be derived read-only from existing responsibility ownership, artifacts, diagnostics, and preserved boundaries. It should not be maintained in a static goal-dimension registry at this time.

Currently supportable dimensions include Operational Responsibility, Execution Visibility, Observation-Derived Capability, selected repository-artifact authorities, Orientation Source Specialization, question-family answer responsibility in a narrow surface-local sense, and pressure visibility in a guarded sense. Unsupported areas include universal ontology, static registry, pressure association, bounded-goal creation, inquiry movement, priority, scheduling, authorization, and execution.

Responsibility family to goal dimension projection audit complete.
