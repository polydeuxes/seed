# Standing-compression recovery pass 091 — fact-index consumer road

## Selected pressure and repository selection

Repository evidence selected the strongest unresolved pressure as concrete production-consumer evidence for the derived fact index, not a new Book clause, universal cache doctrine, or replacement metadata request. The current frontier states that after the cache-boundary reality correction, high-resolution cache testimony vocabulary is no longer current implementation standing and the remaining frontier pressure is concrete production consumer evidence. The implementation then exposes multiple consumers that accept a `DerivedFactIndex` only as an optional acceleration or lookup road while preserving projected-State authority.

This pressure is stronger than adjacent owner-map or universal-topology Unknowns because it has executable witnesses in current code and tests: `capability_candidates`, `capability_verification`, and `capability_promotion_readiness` consume a fact index argument directly, while the current-facts CLI path also uses the fact index for formatting. The pass therefore selects the fact-index-to-capability-consumer road.

## Formation responsibility

Formation responsibility remains with fact-index construction and cache loading. `load_or_build_fact_index(...)` first attempts to load an eligible cached derived-index snapshot, converts it to a `DerivedFactIndex` only after eligibility succeeds, otherwise builds from read-model construction inputs and may save a new derived-index payload with source projection and own boundary metadata. Fact-index formation does not own capability verification, promotion readiness, cluster mutation, or new truth establishment.

## Consumer responsibility

The consumer responsibility is local use of an optional `DerivedFactIndex` as a lookup surface for projected facts. `build_capability_candidates(...)` accepts `fact_index: DerivedFactIndex | None`; without it, the consumer scans `state.facts`, and with it, the consumer enumerates indexed subject/predicate entries and retrieves current projected facts from the same `State`. `build_capability_verification_inspection(...)` and `build_capability_promotion_readiness_inspection(...)` consume the candidate inspection produced from that bounded lookup and preserve their own inspection-only boundaries.

## Realization-language compression

The realization-language compression is:

```text
SQLite/Python derived fact-index cache hit or rebuild
→ DerivedFactIndex lookup object
→ package_installed candidate lookup
→ capability verification or promotion-readiness inspection
```

The compression is lawful only if the index remains a disposable derived lookup over the projected `State`, not a source of fact truth, verification authority, promotion authority, permission, execution, ledger write, or cluster mutation.

## Standing entering and leaving the seam

Entering standing:

- State projection authority and event horizon are inherited from the projected `State` and projection-store eligibility checks.
- Derived fact-index standing is lookup/materialization standing only.
- Read-only and `mutates_cluster=false` boundary standing must survive both cache eligibility and consumer inspection.

Leaving standing:

- Capability-candidate output may say a package-observed candidate is supported by observed package evidence.
- Capability-verification output may say whether candidate verification status is present in existing capability inventory semantics.
- Promotion-readiness output may say whether candidate support and verification support are jointly present for future promotion consideration.
- None of these outputs promotes a package observation into capability proof, permission, selection, planning, execution, ledger mutation, or cluster truth.

## Consumer reliance

The consumers rely on the fact index for lookup completeness and performance over existing projected facts. They do not rely on the fact index for constitutional authority. Their reliance remains bounded because the indexed path and non-indexed path are required to produce equivalent output for the inspected surfaces, and because the consumers pass the indexed facts back through existing candidate and verification evidence formation.

## Fidelity finding

Faithful asymmetry. The producer has stronger cache/formation responsibilities than the consumers need: schema, identity, version, event horizon, payload materialization, source projection boundary, and own non-mutating boundary. The consumers require only a current-fact lookup road over the supplied projected `State`. Tests establish that the indexed path preserves output equivalence and does not mutate state for capability verification and promotion-readiness consumers. This is faithful because the consumer does not treat cache row standing as fact truth or capability authority.

The public CLI behavior remains unchanged.
