## Dimensional findings

- **Identity:** Python IDs are produced by runtime helpers and objects; SQLite preserves IDs in ledger/cache rows. ID equality does not prove constitutional production.
- **Content:** Python forms state/read-model payloads. SQLite stores JSON for selected durable or reusable boundaries. JSON preservation is not establishment.
- **Standing:** Python acts can produce bounded artifact standing; SQLite rows can preserve standing labels only where the boundary says so. Row existence is not represented truth.
- **Source/provenance:** Event payloads and witness tables carry source/provenance fields. SQLite constraints do not create warrant.
- **Responsibility:** Python currently owns most responsible acts. SQLite owns durable preservation and witness-local relational checks.
- **Authority/warrant:** Authority is not added by insertion, query evaluation, foreign keys, or rehydration.
- **Scope/locality:** Workspace IDs, projection names, versions, witness fixture boundaries, and registered-view names scope the realization.
- **Occurrence/preservation:** Event append can evidence a recording occurrence; projection snapshot save evidences preservation of a derived read model only; query evaluation does not evidence uptake.

## Orientation conclusion

The hypothesis is partly supported: event ledgers and projection caches already cross Python and SQLite, but only some crossings preserve explicit standing and occurrence limits. The projection snapshot cache is the smallest evidenced seam where Python produces a derived artifact, SQLite durably preserves it, and Python later consumes it, while prior schema columns did not explicitly preserve that the row is a read-model cache and not a renewed act, fact, warrant, or cluster mutation.
