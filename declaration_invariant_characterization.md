# Declaration Invariant Characterization

## Bounded question

```text
Declaration,

what remains invariant
regardless of where

```

This characterization uses recurring implementation evidence only. It does not recover implementation, recover ownership, recommend slices, authorize campaigns, merge competencies, or introduce a framework.

Repository authority wins.

## Evidence basis

The reviewed evidence is the recurring Declaration evidence already preserved in the repository and app-visible operational surfaces:

- Diagnostic inventory declarations expose named diagnostic surfaces with CLI flags, JSON support, record support, record scope, event-ledger behavior, cluster mutation behavior, diagnostic fact behavior, projected-state use, repository-file use, and descriptions.
- Diagnostic shape audit rows compare declared fields with observed implementation evidence and preserve declared, observed, and status fields.
- Operational surface inventory discovers public CLI options from argparse evidence and classifies whether each surface is registered and JSON-capable.
- Existing Declaration competency interrogation records Declaration as recurring across diagnostic inventory, diagnostic shape audit, operational CLI surfaces, capability verification, responsibility and authority diagnostics, compatibility declarations, and negative-authority declarations.

## Invariant identity

For something to still be Declaration, it must make an implementation-backed claim explicit as an inspectable repository object before another surface relies on that claim.

What cannot change:

- It must name a bounded thing: a surface, diagnostic, candidate, verification status, record scope, JSON surface, boundary, authority diagnostic, or comparable implementation-visible subject.
- It must attach explicit bounded fields to that named thing: flags, booleans, scope labels, implementation paths, function names, markers, limitation strings, boundary notes, unknown values, or status fields.
- It must preserve those fields in inspectable form: registry rows, specs, parser declarations, result objects, JSON dictionaries, rendered reports, or tests.
- It must remain descriptive before reliance. It can make a claim inspectable; it does not become the execution, mutation, promotion, selection, acquisition, permission, policy, or ownership mechanism.
- It must allow later comparison, rendering, or assertion by adjacent competencies.

If the activity stops naming explicit bounded claims, stops preserving them as inspectable material, or becomes the actor that executes, mutates, promotes, selects, acquires, permits, evaluates policy, or owns implementation, it is no longer Declaration under the reviewed evidence.

## Invariant inputs

What must always already exist before Declaration can occur:

- An implementation-visible subject to name.
- A bounded claim shape that can be made explicit.
- Some local authority source for the claim: argparse option evidence, registry fields, implementation spec fields, inspection-result fields, test expectations, output-shape fields, or boundary notes.
- A preservation surface that can carry the claim: dataclass, tuple entry, parser action, JSON dictionary, report row, audit row, or test assertion.
- A boundary between declared claim and later reliance, so another path can inspect, compare, render, or assert the declaration.

What never enters Declaration:

- Runtime execution authority.
- Cluster mutation authority.
- Capability acquisition authority.
- Capability selection authority.
- Policy evaluation authority.
- Permission creation authority.
- Ownership recovery authority.
- Implementation recovery authority.
- Campaign or slice authorization.
- Truth evaluation beyond the local act of making the claim explicit for later audit or comparison.

Declaration can receive implementation evidence and boundary vocabulary. It does not receive authority to turn those inputs into action, mutation, ownership, or recovered architecture.

## Invariant outputs

What must always leave Declaration:

- An explicit named claim preserved in inspectable repository form.
- Fields that another implementation path can inspect, compare, render, or test.
- Positive shape claims when supported, such as supported flags, JSON support, record support, record scope, implementation paths, and output-shape fields.
- Negative authority claims when supported, such as no promotion, no acquisition, no capability selection, no policy evaluation, no tool execution, no event-ledger write, or no cluster mutation.
- Unknowns or limitations when the implementation evidence does not authorize a stronger claim.

What never leaves Declaration:

- Executed work.
- Mutated cluster truth.
- Acquired capability.
- Selected capability.
- Created permission.
- Evaluated policy decision.
- Recovered ownership.
- Recovered implementation.
- Authorized slice or campaign.
- A universal shared framework claim.

Declaration may emit a registry row, implementation spec, audit row, CLI inventory classification, JSON dictionary, rendered report, or test-preserved boundary. It does not emit operational action or recovered ownership.

Recurring negative authority recovered from implementation-backed evidence:

- Declaration must never own execution.
- Declaration must never own cluster mutation.
- Declaration must never own event-ledger mutation unless the specific declared diagnostic surface says event-ledger writing is supported; even then, Declaration owns the declaration of that behavior, not the mutation mechanism.
- Declaration must never own capability acquisition.
- Declaration must never own capability selection.
- Declaration must never own capability promotion.
- Declaration must never own capability-verified fact creation.
- Declaration must never own policy evaluation.
- Declaration must never own permission creation.
- Declaration must never own tool invocation.
- Declaration must never own inventory modification beyond preserving declared inventory entries where that is the local implementation mechanism.
- Declaration must never own diagnostic fact promotion into cluster truth.
- Declaration must never own implementation recovery.
- Declaration must never own ownership recovery.
- Declaration must never own slice recommendation, campaign authorization, or future planning.
- Declaration must never own repository-wide architectural promotion from repeated vocabulary alone.
