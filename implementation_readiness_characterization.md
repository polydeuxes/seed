## Executive answer

Architectural understanding becomes implementation only when repository evidence has reduced an architectural question to a bounded, owned, evidence-backed change whose behavior can be preserved or checked.

The recurring discipline is not:

```text
understanding accumulated
    -> implement
```

The recurring discipline is closer to:

```text
observation / pressure / candidate
    -> bounded inquiry
    -> recurring implementation-backed boundary
    -> explicit owner or corridor
    -> smallest behavior-preserving slice
    -> tests, diagnostics, or shape checks that preserve the boundary
```

Implementation is earned when it is necessary to preserve, expose, test, or relieve a repository-real boundary that has already been recovered. It is not earned by elegance, operator excitement, vocabulary recurrence, accumulated reports, or the feeling that discovery has gone on long enough.

### Signal synthesis

The recurring readiness pattern requires several signals to converge. The strongest convergence is:

```text
independently recurring boundary
+ stable owner or corridor
+ constitutional or implementation necessity
+ smallest behavior-preserving change
+ testable preservation surface
```

A single signal is not enough. Recurrence without boundary stability remains observation. Pressure without implementation necessity remains pressure. Methodology stability without operational need remains methodology.

## Implementation promotion characterization

The repository does demonstrate an implicit implementation promotion process, but not as one universal lifecycle. The process is distributed and family-specific.

Recovered stages, stated conservatively:

1. **Observation or pressure.** Something becomes visible: a diagnostic gap, operator pain, implementation compression, repeated document shape, candidate boundary, or missing visibility.
2. **Bounded unknown.** The repository does not jump to solution. It asks what is unknown, who owns the evidence, and what must not be inferred.
3. **Inquiry or characterization.** Architectural work recovers identity, authority, boundaries, evidence, non-authority, and stop conditions.
4. **Boundary recovery.** The work identifies a smaller implementation-real distinction: observation versus fact, candidate versus verified, visibility versus authority, relationship syntax versus behavior, registry declaration versus execution, structural recurrence versus meaning.
5. **Recurrence and reconciliation.** The boundary is checked across independent cases. Where recurrence is weak, the subject remains a candidate or working methodology.
6. **Implementation necessity test.** The question becomes whether preserving the recovered boundary requires code, tests, CLI surface, diagnostic registration, shape audit, or fixture coverage.
7. **Smallest slice.** If implementation is earned, the slice is local, compatibility-preserving, and test-backed. It does not use the recovery as permission to redesign the organism.
8. **Lawful stop.** If implementation is not earned, the result remains a characterization, reconciliation, pressure record, frontier, or typed unknown.

This process is similar to the Constitution's knowledge-transition discipline: Null persists until evidence justifies movement; evidence moves Null; authority must be explicit; stop is positive.

## Methodology comparison

Implementation promotion and methodology promotion appear to follow a partially shared constitutional discipline.

They share:

- evidence before movement;
- recurrence over isolated vocabulary;
- explicit authority boundaries;
- refusal of attractive but unsupported expansion;
- typed unknowns;
- lawful termination;
- confidence calibration.

They differ in the final promotion target.

Methodology promotion earns a disciplined way to ask or structure future inquiry. It may stabilize report sections, refusal criteria, evidence ledgers, or boundary questions. It does not by itself change runtime behavior.

Implementation promotion must additionally prove implementation necessity: a concrete owner or corridor, behavior to preserve or expose, a smallest change, and tests or diagnostics that keep the boundary from disappearing.

So the comparison is:

```text
partially shared discipline
```

Methodology and implementation share constitutional admission rules, but implementation has a stricter final gate because it mutates repository behavior or preservation surfaces.

## Pressure analysis

The distinction has independently emerged:

```text
Implementation pressure
!=
Implementation readiness
```

Pressure is a lawful input to inquiry. It identifies attention, pain, compression, missing visibility, or unresolved concern. It can be strong evidence that a question matters.

Readiness is a stronger condition. It requires that repository evidence has recovered a bounded implementation-real change that is necessary, owned, and checkable.

The operator report about slow `seed --state-build` is a useful pattern: operator pressure was valid, but instrumentation had to transform it into implementation-backed understanding before any bounded work direction could be trusted. Likewise, surviving pressure after decomposition remains evidence for further observation, not implementation authority.

Therefore pressure may start the readiness investigation, but it cannot finish it.

## Smallest truthful answer

The smallest truthful reason to implement anything is:

```text
A repository-real boundary has recurred strongly enough that leaving it only in operator memory or prose would lose, blur, or make untestable a constitutional distinction that the implementation must preserve.
```

Shorter:

```text
Implement to preserve a necessary, recurring, owned boundary.
```

Not:

```text
Implement because architecture has accumulated.
```

Not:

```text
Implement because pressure feels high.
```

Not:

```text
Implement because the vocabulary is elegant.
```

## When implementation has been earned

Implementation has been earned when all of the following are true:

1. A bounded subject has moved out of Null through repository evidence.
2. The boundary recurs independently enough to avoid one-off promotion.
3. The owner, corridor, or surface is implementation-real or can be made visible without inventing authority.
4. The change preserves or exposes an existing constitutional distinction.
5. The slice can be small, compatibility-preserving, and test-backed.
6. The implementation does not depend primarily on aspiration, convenience, or operator enthusiasm.
7. The lawful stop condition is known if evidence runs out.

## When implementation has not been earned

Implementation has not been earned when the strongest support is any of:

- pressure alone;
- accumulated reports;
- one occurrence;
- elegant or repeated vocabulary;
- operator excitement;
- unowned responsibility;
- unclear mutation boundary;
- methodology that is still changing;
- implementation convenience;
- frontier preservation;
- architecture by aspiration;
- a desire to make discovery feel complete.

In those cases the lawful output is characterization, reconciliation, pressure preservation, typed unknown, or explicit stop.

## Lawful termination

This characterization does not recommend a new implementation slice.

The investigation found that implementation readiness has earned a constitutional characterization, but that characterization is a promotion discipline, not a work queue. It authorizes future refusal as much as future implementation.

The lawful stopping point is:

```text
Implementation is earned by necessary preservation of a recurring, owned, testable boundary.
Pressure may inform that inquiry.
Pressure does not command the implementation.
```
