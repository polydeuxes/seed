# Deep Corpus Examination Capability Topology Audit 001

## 1. Bounded question

Does Seed already possess the implementation-backed owners required to conduct an operator-authorized deep examination over a bounded corpus, and what is the first missing owner that currently forces Codex or a campaign author to conduct the examination instead?

## 2. Vocabulary guardrail

"Operator-authorized" is used only as: the operator supplied or approved the inquiry, scope, permitted operations, and examination resolution. No special operator identity, PrimeOperator, super-operator, epistemic administrator, or constitutional authority class was found or used.

## 3. Operator testimony

The supplied testimony was treated as orientation only: Seed should be able to receive an authorized body of material, examine unfamiliar substrates, recover/request tools, project structure, cross-examine in scope, preserve recurrences/discrepancies/Unknowns, and continue until the authorized frontier is exhausted, suspended, blocked, or stopped. The audit did not treat the proposed ten-step sequence as prescribed implementation.

## 4. Governing distinctions

Preserved distinctions: deep examination is not unbounded execution; authorized corpus is not all accessible data; examining everything in scope is not all-pairs comparison; progressive examination is not incomplete examination; candidate neighborhood is not semantic correspondence; frontier is not priority; projection availability is not sufficiency; missing tool is not missing evidence; external grammar assistance is not Seed constitutional authority; external interpretation is not Seed understanding; campaign completion is not truth completeness; no scheduled work is not no Unknowns; re-examination is not mutation of prior evidence; corpus coverage is not every interpretation; archival completeness is not all-pairs computation; operator-selected depth is not special operator identity.

## 5. Starting evidence

### Constitutional inquiry road

`seed_runtime/bounded_constitutional_question.py` states that it owns only the Operator Inquiry to `BoundedConstitutionalQuestion` boundary and explicitly refuses view projection, capability discovery, selection, ledger writes, and mutation. `BoundedConstitutionalQuestion` preserves caller-supplied inquiry, provenance, bounded question, constitutional intent, scope status, uncertainty, unknowns, and caller fields, but not corpus membership, depth policy, resource policy, or continuation state. `seed_runtime/constitutional_view_selection.py` projects exact caller-declared selection keys, projects registered constitutional view capability keys, selects by exact key comparison, and refuses semantic reasoning, planning, orchestration, evidence discovery, recovery, persistence, and mutation. `seed_runtime/constitutional_pipeline.py` composes those artifacts into a bounded result and provenance explanation. The pipeline stops at selected constitutional views and bounded explanation.

### External website road

`campaigns/project_gutenberg_external_orientation_campaign_001.py` is explicitly a reference scaffold, not canonical website architecture, crawler, or permanent competency. It bounds live requests, bytes, pages, renditions, and delay. In live mode it stops before catalog acquisition unless a sanctioned route exists, and on exceptions it preserves an environment/repository insufficiency classification with Unknowns. `seed_runtime/external_site_rule_testimony.py` preserves caller-supplied site-rule testimony as attributed testimony and explicitly does not determine permission, applicability, precedence, or select/authorize movement.

Known limitation preserved: the live Project Gutenberg website encounter did not supply real site bytes because the execution environment's HTTPS tunnel failed before contact. This audit does not claim Seed ingested or understood the live website.

### Repository comparison road

Tests and campaign evidence show `AGENTS.md` was passed through the same neutral external-material structural and surface-feature projection contract as the selected lesson. The shared projection contract proves mechanical compatibility only. It does not establish semantic correspondence between repository instructions and the selected lesson.

## 12. Inquiry ownership analysis

Current inquiry artifacts preserve question identity, operator inquiry testimony, provenance, bounded question text, constitutional intent, scope status, uncertainty, unknowns, explicit caller-supplied fields, and exact selection keys. They do not preserve depth/examination policy, corpus scope, resource constraints, stop-after-answer versus continue-after-answer, suspension policy, or campaign progress.

Answers:

- Existing inquiry pipeline cannot represent deep examination beyond first sufficiency except as raw operator prose or unsupported caller fields.
- It cannot represent an authorized corpus as first-class membership.
- It cannot represent examination depth as an implementation-owned policy.
- It cannot represent "inspect every eligible artifact under every compatible projection".
- It distinguishes exact constitutional view selection from unsupported keys, but not answer-oriented inquiry from corpus-examination inquiry.
- The current distinction is not constitutional in implementation; deep examination is currently absent as a typed campaign-level handoff.

## 13. Deep-examination representation analysis

Deep examination is not already expressible through the existing constitutional pipeline. The pipeline is intentionally bounded, exact-key, read-only, and non-planning. Treating deep examination as repeated bounded inquiry would still leave unowned corpus membership, compatible projection enumeration, frontier state, next-question generation, re-examination, and completion/suspension. Current evidence supports a campaign-level artifact or handoff, not a new constitutional question family as the first missing boundary.

## 14. Corpus-scope analysis

Seed currently has substrate-specific scope artifacts: external material manifests for selected book excerpts, site scope strings in site-rule testimony, repository path/diagnostic surfaces, and campaign-local constants. These do not compose into one bounded corpus object with included/excluded identities, scope rules, provenance requirements, access status, unknown membership, and authorization evidence.

- Corpus scope exists only partially through manifests, observations, repository roots, diagnostics, and campaign inputs.
- Membership is usually implied by what a script reads or by campaign constants, not explicit campaign membership.
- One campaign containing website, book, and repository material is not represented as a typed Seed artifact.
- Membership changes on discovery are not modeled.
- Authorization/permitted access remains caller- or policy-surface supplied; site-rule testimony explicitly does not authorize movement.
- Caller still supplies material selection and scope boundaries.

## 16. Projection-compatibility analysis

Compatibility is explicit only in narrow places: constitutional selection compares exact question keys to exact view capability keys; external material surface features consume a structural projection type; structural projection validates a manifest/request/text/hash; tool registry can list tools for a capability. There is no general artifact that asks, for each corpus artifact, which projections are compatible, authorized, unavailable, unsupported, blocked, or not-yet-realized.

Capability inventory distinguishes requested/catalog/provider/registered/verified/stale states, and execution separates registered operation validation, policy authorization, execution, recording, and extraction. However, capability projection does not enumerate compatible projections for arbitrary artifacts. Seed can preserve capability needs and resolve registered operation candidates, but it does not produce missing-tool requests from corpus examination gaps nor reapply recovered tools to previously inventoried artifacts.

## 23. Progressive-completeness analysis
