# Selection Warrant Observability District Survey 001

## Scope and governing method

This is a bounded, documentation-only selection and warrant observability survey. It does not implement anything, does not create selector views, does not design a universal `Selector`, warrant engine, observability schema, shared base artifact, or implementation seam, and does not treat absence as defect.

Method consumed from completed Book grammar:

```text
constitutional grammar
→ bounded expectation
→ implementation witness
→ faithful within scope / unfaithful crossing / Unknown
```

The method is not reversed into implementation absence becoming constitutional requirement. A Fidelity crossing is recorded only where established constitutional grammar expects a relation and an implementation witness fails to preserve it.

## Book expectations consumed

The survey consumes these Book expectations as constitutional grammar rather than implementation topology:

- `available != applicable != admitted != consumed != selected != sufficient != warranted != authorized != executed`.
- `support != sufficiency != warrant != authority`.
- `candidate warrant != selection warrant`.
- `selection standing != downstream reliance warrant`.
- `producer warrant != consumer-local warrant`.
- `movement warrant != execution authority`.
- `stopping warrant != completion warrant`.
- Selection artifacts are not selection acts; selected values are not authorization; selection does not itself execute or open inquiry unless the responsible boundary explicitly does so.
- Local warrant families exist without becoming one universal warrant form.
- Realization may be predominantly Python, predominantly SQL, Python-SQL hybrid, responsibility-distributed/native hybrid, or contrastive; one realization's artifact topology is not required in another.

## Per-selector constitutional coordinates

### 1. Selection Path Audit: current focus / pressure category

- **Responsible owner:** `seed_runtime.selection_path_audit`.
- **Bounded question answered:** What implemented selection evidence explains a target such as `current_focus`, `primary_pressure`, or a pressure category?
- **Candidate-set identity:** Pressure audit outputs, ranked as candidates.
- **Visibility:** Pressure candidates are read from `build_pressure_audit`; focus is read from `build_operational_story`.
- **Applicability / eligibility:** Target must match focus selection or pressure category; otherwise unsupported.
- **Admission boundary:** Implemented target match only.
- **Selection evidence / comparison / policy:** Pressure audit ordering by descending score, then category name.
- **Selected result:** First pressure item for pressure/focus selections, or `unknown` for unsupported target.
- **Lawful non-selection:** Unsupported target and empty candidate set preserve Unknown rather than invent selection.
- **Non-selected remainder:** Preserved as pressure candidates after selected item, with score/tie reasons.
- **Refusal / conflict / Unknown:** Unsupported target records implementation Unknown; no candidates records candidate-set evidence gap.
- **Standing produced:** Read-only audit/explanation of implemented selection evidence.
- **Stronger standing not produced:** Does not prove arbitrary repository selection logic, record facts, write ledger, mutate cluster, authorize, execute, or establish downstream reliance.

### 2. Constitutional View Selection

- **Responsible owner:** `seed_runtime.constitutional_view_selection`.
- **Bounded question answered:** Which registered constitutional read-model view names match exact caller-declared selection keys for one bounded question projection?
- **Candidate-set identity:** Registered constitutional read-model capability projections.
- **Visibility:** Question projection carries exact selection keys; capability projection regenerates registered source capability keys.
- **Applicability / eligibility:** Exact intersection between question keys and capability keys.
- **Admission boundary:** Registered view names only; projections only.
- **Selection evidence / comparison / policy:** Deterministic exact-key comparison, no semantics/ranking/heuristics.
- **Selected result:** Tuple of registered view names.
- **Lawful non-selection:** No matched registered view appends uncertainty instead of forcing a view.
- **Non-selected remainder:** Not preserved as full candidate remainder; faithful within scope because composition consumes selected registered names and unsupported-key uncertainty, not full candidate universe.
- **Refusal / conflict / Unknown:** Unsupported selection keys and no-match uncertainty are preserved.
- **Standing produced:** Representation-selection artifact for composition request formation.
- **Stronger standing not produced:** No consideration selection, constitutional recovery, evidence discovery, orchestration, repository mutation, ledger write, cluster mutation, authorization, or proof of exact-key producer occurrence when direct request construction bypasses producer.

### 3. Advancement-Need Consideration Selection

- **Responsible owner:** `seed_runtime.advancement_need_consideration_selection`.
- **Bounded question answered:** Does explicit focus evidence name exactly one visible selectable advancement-need reference for consideration?
- **Candidate-set identity:** One advancement-need reference set with need-set, selected-goal, horizon, family, native projection, and native lineage identities.
- **Visibility:** Visible references come from the supplied reference set; focus evidence supplies exact reference identity and lineage fields.
- **Applicability / eligibility:** Exact identity/lineage match, selectable reference, no conflict/ambiguity/missing identity.
- **Admission boundary:** Focus evidence must be exact and match the same reference set and native lineage.
- **Selection evidence / comparison / policy:** Exact reference focus evidence; uniqueness alone does not select.
- **Selected result:** One selected reference when state is `selected`.
- **Lawful non-selection:** `no_focus_evidence`, `missing_identity`, `ambiguous`, `conflict`, `reference_mismatch`, `absent_reference`, `duplicate_lineage_conflict`, and `non_selectable`.
- **Non-selected remainder:** Visible and non-selected references are preserved.
- **Refusal / conflict / Unknown:** Missing identity, ambiguity, conflicts, absent references, duplicate lineage, and non-selectable states are explicit.
- **Standing produced:** Need selected for consideration.
- **Stronger standing not produced:** Not highest-priority need, primary blocker, resolution, next action, realization selection, inquiry opened, authority requested, authorization, execution, recording, event-ledger write, or mutation.

### 4. Contextual Interpretation Warrant Set and Meaning Selection

- **Responsible owner:** `seed_runtime.contextual_interpretation_warrant_set` for candidate-local warrants; `seed_runtime.contextual_interpretation_selection` for selection.
- **Bounded question answered:** First, what standing does each interpretation candidate have from explicit retrospective/clarification evidence? Second, does explicit candidate-bound selection evidence select one warranted interpretation?
- **Candidate-set identity:** Supplied interpretation candidates over exact operator material/source spans.
- **Visibility:** Candidate warrants preserve source spans, examined evidence by disposition, corrections, residual source material, unknowns, conflicts, and known loss.
- **Applicability / eligibility:** Selection evidence must reference exactly one known candidate with `warranted` standing.
- **Admission boundary:** Unknown/unwarranted candidate references, conflicting selection evidence, and unwarranted candidate selection evidence are refused.
- **Selection evidence / comparison / policy:** Explicit `exact_operator_clarification` or `candidate_bound_selection_artifact`; a unique warranted candidate is not automatically selected.
- **Selected result:** One selected candidate reference and candidate if evidence is valid.
- **Lawful non-selection:** No selection evidence, warranted candidate without selection evidence, multiple warranted candidates without selection evidence, selection refused for unknown/unwarranted candidate, conflicting selection evidence.
- **Non-selected remainder:** Non-selected candidate warrants are preserved.
- **Refusal / conflict / Unknown:** Candidate unknowns/conflicts and selection conflicts survive.
- **Standing produced:** Selected interpretation standing only when explicit candidate-bound evidence selects a warranted candidate.
- **Stronger standing not produced:** No downstream applicability, admission, goal binding, inquiry movement, authorization, execution, recording, conversation/state/cluster mutation.

### 5. Closed-Choice Selection Binding

- **Responsible owner:** `seed_runtime.closed_choice_selection_binding`.
- **Bounded question answered:** Does a captured operator token bind to one option inside the exact presented closed choice set?
- **Candidate-set identity:** Presented closed choice set and fingerprint.
- **Visibility:** Presented prompt/options and token capture.
- **Applicability / eligibility:** Token must belong to the exact choice set and not be defeated by conflicts/unknowns.
- **Admission boundary:** Token capture must reference same choice set; otherwise error.
- **Selection evidence / comparison / policy:** Exact token lookup in presented options.
- **Selected result:** Bound option ref/label when state is `bound`.
- **Lawful non-selection:** `unsupported`, `unknown`, or `conflict` states.
- **Non-selected remainder:** Presented options are preserved; non-selected reason per option is not separately enumerated.
- **Refusal / conflict / Unknown:** Unsupported token, unknown selection evidence, and conflicting selection evidence are preserved.
- **Standing produced:** Local binding of token to option.
- **Stronger standing not produced:** No goal transition, inquiry frontier transition, operator authority, execution authorization, or application to goal.

### 6. Examination-Work Selection

- **Responsible owner:** `seed_runtime.examination_work_selection`.
- **Bounded question answered:** Under an existing policy projection and frontier, is at most one eligible examination work item selected?
- **Candidate-set identity:** Examination frontier work items, policy projection, and selector handoff.
- **Visibility:** Eligible, blocked, unsupported, deferred, examined, failed, conflict/unknown classifications remain frontier-owned; policy projection supplies scope, sufficiency, prerequisites, and no-selection conditions.
- **Applicability / eligibility:** Policy must be applicable and sufficient; eligible references must match frontier eligibility.
- **Admission boundary:** Handoff and policy identities must match the supplied frontier; policy references must exist in frontier.
- **Selection evidence / comparison / policy:** Explicit work identity, sole eligible no-order policy, prerequisite-first policy, explicit no-selection, and refusal of arbitrary tie-breaks.
- **Selected result:** One work reference when state is `selected`.
- **Lawful non-selection:** Conflict/unknown policy, inapplicable policy, no-selection policy, no eligible work, multiple eligible without lawful tie-break, named work absent or ineligible, unsupported policy kind.
- **Non-selected remainder:** Non-selected eligible work is preserved with reasons; non-eligible classifications are preserved unchanged.
- **Refusal / conflict / Unknown:** Conflict, unknown, invalid identity, and no-selection reasons are explicit.
- **Standing produced:** Selection standing for future probe-request handoff only when selected.
- **Stronger standing not produced:** Does not create/alter policy, authorize, schedule, execute, mark examined, or declare campaign completion.

### 7. Operational-Realization Selection

- **Responsible owner:** `seed_runtime.operational_realization_selection`.
- **Bounded question answered:** Which zero or one supported operational realization candidate is selected under reachability and policy?
- **Candidate-set identity:** Candidate set, reachability projection, future selection handoff, capability demand, and probe request.
- **Visibility:** Supporting, blocked, unsupported, unknown, and conflicting candidate partitions are preserved from reachability.
- **Applicability / eligibility:** Reachability must be `reachable`; candidates must be supporting; policy may be exact candidate, sole supported candidate, select none, constraints, or insufficient.
- **Admission boundary:** Validation checks handoff, candidate set, reachability projection, probe request, capability demand, candidate partition, and candidate standing identities.
- **Selection evidence / comparison / policy:** Explicit policy; candidate ordering does not determine selection.
- **Selected result:** Zero or one selected candidate reference.
- **Lawful non-selection:** Reachability not reachable, insufficient policy, select-none policy, required candidate not eligible, no supporting candidate, unresolved tie, no candidate satisfies policy, conflict.
- **Non-selected remainder:** Non-selected supporting candidates and alternative candidate references are preserved.
- **Refusal / conflict / Unknown:** Policy conflicts, policy unknowns, unknown/conflicting candidate partitions, and unresolved ties survive.
- **Standing produced:** Operational-realization selection standing and, if selected, a future warrant handoff.
- **Stronger standing not produced:** Selection explicitly does not warrant reliance, construct invocation request, translate meaning, authorize, schedule, or execute.

### 8. Operational-Realization Warrant

- **Responsible owner:** `seed_runtime.operational_realization_warrant`.
- **Bounded question answered:** May one exact selected operational realization be relied on for the exact demand/candidate/grammar/contract/support/current state?
- **Candidate-set identity:** Consumes selection, future warrant handoff, reachability projection, candidate set, and selected candidate.
- **Visibility:** Preserves grammar, behavior, representation, methodology, mechanism/dependency/authority standings, supporting basis, limitations/loss, provenance, unknowns, conflicts, and non-selected alternatives.
- **Applicability / eligibility:** Selection must be selected; selected candidate must match handoff and candidate set; reachability must be reachable; candidate must be supporting/eligible/read-only.
- **Admission boundary:** Extensive identity/provenance validation; mismatch raises refusal error.
- **Selection evidence / comparison / policy:** Not a selector; consumes selected candidate standing and evaluates warrant state.
- **Selected result:** Not applicable; warranted/refused reliance state.
- **Lawful non-selection:** Not applicable.
- **Refusal / conflict / Unknown:** Conflict, insufficient support, unknown standing, missing support/provenance/contract/grammar, unavailable authority/dependency/mechanism are preserved.
- **Standing produced:** `warranted`, `insufficient`, `unknown`, or `conflict` reliance standing.
- **Stronger standing not produced:** Does not authorize, schedule, emit, execute, construct external representation, create registry warrant, or collapse authority reachability into constitutional authorization.

## Lawful realization asymmetries

- Python dataclasses currently preserve many local distinctions directly: identities, selected/non-selected partitions, Unknowns, conflicts, booleans denying authority, and future handoffs.
- A predominantly SQL realization could lawfully preserve comparable relations as predicates, joins, rows, and query outputs if it preserves bounded subject, owner, support, standing, scope, contrary material, and Unknown; it need not mimic dataclass shape.
- A Python-SQL hybrid could split candidate visibility, policy applicability, and audit presentation across substrates, provided handoff standing and identity are not compressed.
- A responsibility-distributed/native hybrid could place producer and consumer-local assertions in distinct services or records; exact artifact co-location is not constitutionally required.
- A contrastive realization could expose what each substrate compresses, but the contrast would remain evidence, not constitutional grammar by itself.

## Topology discussion

### What selection families appear genuinely distinct?

The survey recovers at least these distinct families:

- **Representation selection:** constitutional view names selected by exact deterministic keys for composition.
- **Consideration/focus selection:** advancement need selected for consideration from explicit focus evidence and lineage checks.
- **Meaning selection:** contextual interpretation selected only after candidate-local warrants and explicit candidate-bound selection evidence.
- **Closed-choice binding:** operator token bound inside one exact presented choice set.
- **Policy/work selection:** examination work selected through policy projection, frontier eligibility, and refusal of arbitrary ties.
- **Operational-realization selection:** zero/one mechanism candidate selected from reachable supporting candidates under policy.
- **Audit/explanation selection visibility:** selection path audit reconstructs implemented focus/pressure evidence without becoming a producer of broader selection standing.

They should not be collapsed into one selector family. They differ in candidate identity, visibility, admission, non-selection, standing produced, consumer reliance, and authority denied.

### Which warrant relations recur without becoming universal?
