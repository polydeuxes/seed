## Independent producer recovery

### Bounded purpose governing focus

A lawful producer, if present, would have to answer a narrower question than selection: what bounded responsibility establishes that exactly one visible selectable advancement-need reference is in focus for present consideration? The repository contains an artifact shape (`NeedFocusEvidence`) capable of carrying that answer, but the inspected runtime does not contain a responsibility that derives it from goal, horizon, policy, priority, dependency, temporal condition, prior inquiry residue, or operational state.

### Subject placed in focus

The subject would be exactly one `AdvancementNeedReference`, not a native need object, goal, family, route, inquiry, or work item. The evidence shape is capable of naming this through `reference_id` plus need-set, selection, goal-establishment, horizon, family, native projection, and native lineage coordinates.

### Candidate reference set visible to the producer

No recovered production owner consumes `AdvancementNeedReferenceSet` to form `NeedFocusEvidence`. The only production owner that consumes the reference set is the downstream selector. `project_advancement_need_reference_set(...)` creates the candidate set but does not name one reference as focused.

### Testimony or standing the producer consumes

No production owner was found that consumes bounded purpose, policy, dependency standing, prior inquiry residue, temporal condition, or authority boundary to choose one `AdvancementNeedReference` as focused. The only standing consumed in production by the consideration selector is already-supplied `NeedFocusEvidence`.

### Applicability and admission boundaries

Because no producer was recovered, no Seed-local applicability/admission boundary for forming focus standing was recovered. What exists is a consumer-side admission boundary: the selector admits exact focus evidence only if it matches the current reference set and a selectable reference.

### Rule or act by which one exact reference becomes focused

No Seed-local focus-production rule was recovered. The repository explicitly refuses several implicit substitutes: uniqueness, sufficiency reasons, standing labels, presentation order, family, wording similarity, severity, and selectable count do not select a need. Those refusals also block treating them as focus-formation rules for this road.

### Unknowns, conflicts, refusals, ambiguity, and absence

The focus artifact can carry `missing_identity`, `ambiguous`, `conflict`, and `Unknown` states. The selector preserves those as non-selection states instead of inventing an exact focus. Absence of evidence becomes `no_focus_evidence`; absent named references, duplicate lineage, mismatched coordinates, and non-selectable references are separately refused.

It is not priority assignment, primary-blocker declaration, resolution selection, route selection, inquiry opening, next-action selection, realization selection, authorization, execution, recording, event-ledger writing, or mutation.

## Independent consumer recovery

### Accepted artifact or testimony

The consumer accepts an `AdvancementNeedReferenceSet` and iterable `NeedFocusEvidence`. An empty iterable is meaningful and produces `no_focus_evidence` rather than an inferred choice.

### Identities that must match

For exact focus evidence to select, all exact evidence items must name the same `reference_id`. Each exact item must match the reference set's `need_set_id`, `selection_id`, `goal_establishment_id`, and `horizon_id`, and must provide `family`, `native_projection_id`, and non-empty `native_lineage`. The matched visible reference must then match family, native projection, and native lineage exactly.

### Standing validated locally

The selector validates that the named reference is visible in the reference set, singular, lineage-consistent, not conflict-marked, and selectable. Selectability itself is inherited from the reference set's established native bucket and established native standing.

### Ambiguity or conflict refused

The consumer refuses explicit conflict evidence, missing or Unknown identity, ambiguous candidate IDs, multiple exact named IDs, absent references, duplicate lineage conflicts, mismatched coordinates, and non-selectable references.

### What makes a reference selectable

The reference producer marks a reference selectable only when the native bucket is `established` and native standing is `established`. Unsupported, unknown, conflicting, excluded, outside-scope, unclassified, and duplicate-lineage records remain visible but non-selectable.

### Selection standing produced

A successful consumer result is `AdvancementNeedConsiderationSelection(selection_state="selected")` with the exact selected reference. It also preserves the full visible reference snapshot, non-selected references, focus evidence refs, provenance refs, unknowns, conflicts, read-only state, and explicit negative authority flags.

### Non-selected references preserved

All visible references except the selected reference are preserved as `non_selected_references`. In non-selected states, the visible references remain visible and no selected reference is asserted.

### What the consumer explicitly refuses to infer

The consumer refuses to infer selection from uniqueness, sufficiency reasons, standing labels, presentation order, family, wording similarity, severity, or selectable count. It also refuses to convert selection-for-consideration into priority, blocker, route, inquiry opening, next action, authorization, execution, recording, event-ledger write, or mutation.
